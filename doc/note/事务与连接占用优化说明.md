# 事务与数据库连接占用优化说明

本文档记录为缓解**长事务**与**连接池占用**所做的代码调整（面向压测与生产稳定性），并附**回归测试要点**。

## 1. 背景与目标

- **问题**：在 `@Transactional` 范围内执行外部 HTTP（支付宝）、RocketMQ 发送、Redis、OpenFeign 等操作时，会长时间占用数据库连接，易导致连接池耗尽、TPS 下降。
- **目标**：
  - 缩短事务边界，尽快 `commit` 释放连接。
  - MQ / 缓存 / RPC 尽量在**事务提交之后**执行（`afterCommit` 或事务外）。
  - 批量写库减少往返，降低事务持有时长。

## 2. 改动清单

### 2.1 作者积分充值下单（高优先级）

| 项 | 说明 |
|----|------|
| **文件** | `novel-user/.../PaymentServiceImpl.java` |
| **原状** | `createRechargeOrder` 整段带 `@Transactional`，在事务内 `insert` 后调用支付宝 `pageExecute`。 |
| **现况** | 使用 `TransactionTemplate.executeWithoutResult` **仅包裹插入待支付订单**；支付宝调用在**短事务提交之后**执行。 |
| **附加** | 先校验支付方式再写库，避免「已插入订单再发现渠道不支持」。 |

### 2.2 书籍 / 章节审核（高优先级）

| 项 | 说明 |
|----|------|
| **文件** | `novel-book/.../BookAuditServiceImpl.java` |
| **原状** | `processChapterAuditResult`、`processBookAuditResult`、`manualAudit` 在事务内执行 Redis 删除、`rocketMQTemplate` 发送、`userFeignManager` 调用等。 |
| **现况** | 新增 `runAfterCommit(Runnable)`；**书籍变更 MQ**、**作者通知（Feign + SSE）**、**相关 Redis 删缓存**在 **事务提交后**执行；`updateBookFromAudit` 中的书籍信息缓存清理同样延后到提交后。 |
| **语义** | 与改前一致：侧翼失败（MQ/Feign/Redis）仍以日志为主，**不回滚**已提交的数据库变更。若需与库强一致投递，需另行引入事务消息或补偿表（本次未做）。 |

### 2.3 章节提交 MQ 消费（中高优先级）

| 项 | 说明 |
|----|------|
| **文件** | `novel-book/.../ChapterSubmitListener.java` |
| **原状** | 在 `@Transactional` 的 `onMessage` 内同步发送：ES 同步、章节审核请求、章节更新通知等 MQ。 |
| **现况** | 新增 `sendRocketMqAfterCommit`，上述 MQ 均在 **`afterCommit`** 发送（无活跃事务时立即发送）。 |

### 2.4 作者积分 MQ 落库（中优先级）

| 项 | 说明 |
|----|------|
| **文件** | `novel-user/.../AuthorPointsConsumeListener.java`、`AuthorPointsRollbackListener.java` |
| **原状** | `onMessage` 带 `@Transactional`，在事务内先读 Redis 再写库。 |
| **现况** | **事务外**读取 Redis 并解析数值；`TransactionTemplate.executeWithoutResult` **仅包裹**写 `author_info` 与插日志。 |

### 2.5 书架订阅章节更新通知（中高优先级）

| 项 | 说明 |
|----|------|
| **文件** | `novel-user/.../BookChangeMqListener.java`；`MessageReceiveMapper.java` + `mapper/MessageReceiveMapper.xml` |
| **原状** | 对每位订阅用户循环 `messageReceiveMapper.insert`。 |
| **现况** | 新增 `insertBatch`，一条 SQL 批量插入 `message_receive`，缩短事务与网络往返。 |

## 3. 涉及的业务功能（便于测试分域）

1. **充值**：作者创建积分充值订单（支付宝 H5/相关流程）。
2. **审核**：AI 审核结果消费、人工审核；通过后 ES 通知、缓存失效、作者站内信/SSE。
3. **章节**：作者通过 MQ 提交/更新章节；ES、AI 审核请求、读者侧章节更新通知。
4. **积分**：消费/回滚 MQ 将 Redis 积分同步到 DB 并记日志。
5. **消息**：多用户加书架时，单书章节更新后的批量站内通知。

## 4. 回归测试建议

### 4.1 充值（`PaymentServiceImpl`）

- 合法作者 + 支付宝渠道：生成待支付订单且能拿到支付链接。
- 不支持渠道：不应写入订单（相对原逻辑的修正点）。
- 支付宝失败：库中可能仍有待支付单，依赖关单/对账（与历史「异常被 catch 仍可能已提交」类问题需产品/运维共识）。

### 4.2 审核（`BookAuditServiceImpl`）

- 章节/书籍 AI 审核通过（高置信）：库表状态正确；缓存刷新；ES 消费侧正常；作者通知与 SSE（若有）。
- 审核不通过：章节内容缓存删除等行为正确。
- 人工审核：书籍分支与章节分支的缓存、MQ、作者通知。
- 关注：**仅当 DB 提交成功后**才会发 MQ / 调用户服务，避免「库回滚但消息已发出」的旧竞态。

### 4.3 章节 MQ（`ChapterSubmitListener`）

- 新建/更新章节；开/关审核开关；书籍已审核通过时 ES 消息是否仍发出。
- 下游：搜索、用户服务「章节更新」订阅通知。

### 4.4 积分 MQ（两个 Listener）

- 扣减与回滚各走一条：DB 与 Redis 一致，幂等键重复时不重复插日志。

### 4.5 书架通知（`BookChangeMqListener`）

- 多用户订阅同一本书，章节更新后每人一条消息，内容与跳转链接正确；无订阅时不报错。

## 5. 后续可选优化（未实施）

- 连接池大小与各服务线程数、MQ 消费并发匹配调优（`spring.datasource.hikari.*`）。
- 需「与库同事务」的 MQ：本地消息表、RocketMQ 事务消息等方案。
- `BookChangeMqListener` 在订阅量极大时，可再考虑分批 `insertBatch` 或调大 `max_allowed_packet`（MySQL）。
