# 作者积分强一致扣减与回滚改造说明

## 一、改造目标

本次改造的核心目标是：**积分余额不再以 Redis 作为强一致来源，MySQL 才是作者积分主账本**。

改造后的职责划分如下：

- `author_info`：保存作者当前积分余额。
- `author_points_tx`：保存积分事务事实，负责强幂等和回滚依据。
- `author_points_consume_log`：保存消费/回滚展示流水，用于统计、审计、后台展示。
- Redis：只作为积分读缓存，不能作为 DB 写入依据。
- RocketMQ：只做异步日志/事件兜底，不再决定余额正确性。

## 二、核心表设计

### 1. 作者余额表：`author_info`

积分余额仍保存在作者表中：

- `free_points`：免费积分余额。
- `paid_points`：付费积分余额。
- `free_points_update_time`：免费积分最近重置/更新时间。

扣减和回滚都会在事务内对该行执行 `select ... for update`，保证同一作者的并发积分变更串行化。

### 2. 积分事务表：`author_points_tx`

该表是新增的积分主事务表，用于强幂等、回滚依据和审计事实。

```sql
CREATE TABLE author_points_tx (
    id bigint unsigned NOT NULL AUTO_INCREMENT COMMENT '主键',
    request_id varchar(128) NOT NULL COMMENT '业务幂等号',
    author_id bigint unsigned NOT NULL COMMENT '作者ID',
    tx_type tinyint unsigned NOT NULL COMMENT '事务类型;1-扣减 2-回滚 3-充值',
    consume_type tinyint unsigned NULL COMMENT '消费类型;0-AI审核 1-AI润色 2-AI封面',
    free_points_delta int NOT NULL DEFAULT 0 COMMENT '免费积分变动值；扣减为负，回滚/充值为正',
    paid_points_delta int NOT NULL DEFAULT 0 COMMENT '付费积分变动值；扣减为负，回滚/充值为正',
    status tinyint unsigned NOT NULL COMMENT '状态;1-成功 2-已回滚',
    related_id bigint unsigned NULL COMMENT '关联ID',
    related_desc varchar(255) NULL COMMENT '关联描述',
    create_time datetime NULL COMMENT '创建时间',
    update_time datetime NULL COMMENT '更新时间',
    PRIMARY KEY (id),
    UNIQUE KEY uk_request_id (request_id),
    KEY idx_author_id (author_id),
    KEY idx_author_create_time (author_id, create_time)
) COMMENT='作者积分事务表';
```

关键点：

- `request_id` 是强幂等核心，同一个业务请求只能成功写入一次。
- 扣减事务使用原始 `requestId`。
- 回滚事务使用 `requestId + ":ROLLBACK"`。
- 扣减成功后 `status = 1`，回滚成功后原扣减事务更新为 `status = 2`。

### 3. 消费日志表：`author_points_consume_log`

该表继续保留，但不再作为主账本，只做展示和统计。

需要保证 `idempotent_key` 长度足够：

```sql
ALTER TABLE author_points_consume_log
    MODIFY COLUMN idempotent_key VARCHAR(128) NULL COMMENT '幂等性Key（MQ消费去重，对应消费requestId）';
```

## 三、扣减流程

入口方法：`AuthorServiceImpl#deductPoints`

扣减现在完全基于 MySQL 本地事务：

1. 解析作者 ID。
2. 解析业务幂等号 `requestId`。
3. 开启 MySQL 事务。
4. 查询 `author_points_tx`：
   - 如果 `request_id` 已存在，说明重复请求，直接幂等成功返回。
5. `select ... for update` 锁住 `author_info` 作者积分行。
6. 再次查询 `author_points_tx`，防止并发请求在等待行锁期间已经完成扣减。
7. 根据 `free_points_update_time` 判断是否需要重置免费积分。
8. 按“先免费积分，后付费积分”的顺序计算扣减额度。
9. 如果积分不足，事务失败，返回余额不足。
10. 更新 `author_info.free_points / paid_points`。
11. 插入 `author_points_tx` 扣减事务：
    - `tx_type = 1`
    - `free_points_delta < 0`
    - `paid_points_delta < 0`
    - `status = 1`
12. 插入 `author_points_consume_log` 展示流水：
    - 免费积分流水 key：`requestId + "_FREE"`
    - 付费积分流水 key：`requestId + "_PAID"`
13. 事务提交后刷新 Redis 最新余额。
14. 发送 RocketMQ 消息，仅作为异步日志/事件兜底。

扣减成功后的事务示例：

```text
request_id = COVER:12:99:abc123
tx_type = 1
free_points_delta = -30
paid_points_delta = -70
status = 1
```

## 四、回滚流程

入口方法：`AuthorServiceImpl#rollbackPoints`

回滚必须基于原扣减 `requestId`，不能重新生成随机 key。

流程如下：

1. 解析作者 ID。
2. 如果本次扣减是重复请求被幂等拦截，则直接返回成功，不执行回滚。
3. 校验请求中必须存在原始 `requestId`。
4. 查询 `author_points_tx` 中是否存在 `requestId + ":ROLLBACK"`：
   - 存在则说明已经回滚过，直接幂等成功。
5. 查询原扣减事务 `requestId`：
   - 不存在则无法回滚。
   - `status = 2` 说明原扣减已经回滚，直接幂等成功。
6. 根据原扣减事务中的负数 delta 计算应退积分。
7. 开启 MySQL 事务。
8. `select ... for update` 锁住 `author_info` 作者积分行。
9. 再次查询回滚事务，防止并发重复回滚。
10. 将免费/付费积分加回 `author_info`。
11. 插入 `author_points_tx` 回滚事务：
    - `request_id = 原requestId + ":ROLLBACK"`
    - `tx_type = 2`
    - `free_points_delta > 0`
    - `paid_points_delta > 0`
    - `status = 1`
12. 将原扣减事务更新为 `status = 2`。
13. 插入一条回滚展示日志到 `author_points_consume_log`。
14. 事务提交后刷新 Redis 最新余额。

回滚事务示例：

```text
request_id = COVER:12:99:abc123:ROLLBACK
tx_type = 2
free_points_delta = 30
paid_points_delta = 70
status = 1
```

## 五、Redis 的定位

Redis 现在只做缓存：

- 扣减成功后：用 MySQL 最新余额刷新 Redis。
- 回滚成功后：用 MySQL 最新余额刷新 Redis。
- 查询作者状态时：从 MySQL 获取余额，必要时在 DB 中重置免费积分，再刷新 Redis。

禁止的做法：

- 不能从 Redis 读取余额后写回 MySQL。
- 不能用 Redis 扣减结果作为 DB 更新依据。
- 不能用 Redis `SETNX` 决定免费积分是否应写 DB 重置。

这样即使 Redis 旧值、丢值、写失败，也不会影响积分主账本正确性。

## 六、RocketMQ 的定位

当前 MQ 消费者已经不再写 `author_info` 余额。

现状：

- `AuthorPointsConsumeListener`：只尝试插入消费展示日志。
- `AuthorPointsRollbackListener`：只尝试插入回滚展示日志。
- 重复消息依赖 `author_points_consume_log.idempotent_key` 唯一索引兜底。

余额正确性不依赖 MQ，因此 MQ 乱序或重投递不会导致重复扣减或重复回滚。

如果后续要补顺序消费，可以再改：

```java
rocketMQTemplate.syncSendOrderly(destination, mqDto, String.valueOf(authorId));
```

Listener 配置：

```java
consumeMode = ConsumeMode.ORDERLY
```

但顺序消费现在只影响异步事件处理顺序，不影响余额正确性。

## 七、幂等策略

### 1. 扣减幂等

同一个业务请求必须使用同一个 `requestId`。

后端通过 `author_points_tx.uk_request_id` 保证：

- 第一次请求：扣减积分并插入事务。
- 重复请求：查到事务已存在，直接返回成功，不重复扣分。

### 2. 回滚幂等

回滚使用原扣减 `requestId` 派生：

```text
rollbackRequestId = requestId + ":ROLLBACK"
```

后端通过 `author_points_tx.uk_request_id` 保证：

- 第一次回滚：加回积分并插入回滚事务。
- 重复回滚：查到回滚事务已存在，直接返回成功，不重复退分。

### 3. AI 审核 requestId 与内容版本

AI 审核和润色、封面不同：章节和书籍内容会被反复修改，同一个章节可能多次触发审核。

因此审核类 `requestId` 不能只使用：

```text
AUDIT_CHAPTER:{authorId}:{chapterId}
AUDIT_BOOK:{authorId}:{bookId}
```

否则作者第一次提交章节审核完成后，再修改章节内容重新审核，仍会命中同一个积分事务，导致新的审核请求被误判为重复请求。

当前方案是在书籍表和章节表增加内容版本号：

- `book_info.version`：书籍版本号，默认 `0`。
- `book_chapter.version`：章节版本号，默认 `0`。

当会影响审核结论的内容发生变化时，业务侧递增版本号：

- 书籍：书名、简介、分类等送审字段变化时，`book_info.version + 1`。
- 章节：章节号、章节名、章节正文变化时，`book_chapter.version + 1`。
- 普通状态变更、缓存刷新、通知发送等不改变审核内容的操作，不递增版本号。

审核请求的幂等号应绑定到“具体内容版本”：

```text
AUDIT_BOOK:{authorId}:{bookId}:v{version}
AUDIT_CHAPTER:{authorId}:{chapterId}:v{version}
```

这样同一版本重复提交会被 `author_points_tx.request_id` 拦截，不会重复扣分；内容修改后版本号变化，会生成新的 `requestId`，可以正常扣分并重新审核。

对于前端新建章节但尚未拿到数据库章节 ID 的同步 AI 审核场景，前端使用章节标题、章节号、正文摘要 hash 参与生成 `requestId`：

```text
AUDIT_CHAPTER:{bookId}:{chapterNum/chapterName}:{contentHash}
```

同一份未保存内容重复点击不会重复扣分，内容发生变化后会得到新的幂等号。

### 4. MQ 审核结果幂等与防旧消息覆盖

审核 MQ 也携带版本号，避免旧审核结果覆盖新内容。

当前链路如下：

1. `BookSubmitListener` / `ChapterSubmitListener` 在内容变化时递增 `version`。
2. 发送审核请求 MQ 时携带当前 `version`。
3. 审核任务 ID 使用稳定格式：

   ```text
   audit_book_{bookId}_v{version}
   audit_chapter_{chapterId}_v{version}
   ```

4. AI 服务处理完成后，审核结果 MQ 原样带回 `version`。
5. `BookAuditServiceImpl` 落库前重新查询当前书籍/章节版本：
   - 当前版本与结果版本一致：允许更新审核状态。
   - 当前版本与结果版本不一致：说明作者已经提交了新内容，旧审核结果直接丢弃。

这个设计解决的是“旧消息覆盖新状态”的问题。例如章节版本 `v1` 正在审核时，作者又修改内容提交 `v2`，如果 `v1` 的结果后到达，系统不会把 `v1` 的审核通过/不通过写到 `v2` 内容上。

## 八、当前仍可继续完善的点

1. **充值链路强幂等**
   支付回调仍建议改为条件更新：

   ```sql
   UPDATE author_points_recharge_log
   SET recharge_status = 1, trade_no = ?, recharge_time = NOW()
   WHERE out_trade_no = ? AND recharge_status = 0;
   ```

   只有影响行数为 1 时，才在同一事务内增加积分并插入 `author_points_tx` 充值事务。

2. **MQ 事件模型简化**
   既然余额已经由 MySQL 主账本保证，后续可以把扣减/回滚 MQ 统一成积分事件消费者，或只保留审计/通知能力。

3. **前端 requestId 保存策略**
   当前前端已传 `requestId`。如果要更严格防止用户双击同一个请求，可以在组件状态中保存本次请求 ID，请求完成后再释放。

## 九、面试表达

可以这样概括：

> 原方案是 Redis 扣减、MQ 异步落库，适合最终一致，但不适合作为积分主账本。改造后我把 MySQL 作为积分主账本，所有扣减和回滚都在数据库本地事务中完成，通过 `author_points_tx.request_id` 唯一索引实现强幂等，通过 `select ... for update` 保证同一作者并发扣减串行化。Redis 只做缓存，事务提交后刷新。MQ 只做异步日志和事件处理，不再决定余额正确性。这样即使接口重试、AI 失败重复回调、MQ 重投递，也不会重复扣减或重复回滚。

如果面试官继续追问“章节修改后重新审核如何避免幂等号冲突”，可以补充：

> 审核类 requestId 不是只绑定章节 ID，而是绑定章节或书籍的内容版本。书籍和章节表都有 `version` 字段，影响审核结论的内容变化时版本号递增，审核请求、审核任务 ID、审核结果消息都会携带该版本。扣积分时用 `AUDIT_CHAPTER:{authorId}:{chapterId}:v{version}` 作为幂等号，同一版本重复提交不会重复扣分；内容更新后版本号变化，会形成新的业务请求。审核结果落库前也会校验结果版本和当前 DB 版本，不一致就丢弃旧结果，防止旧审核消息覆盖新内容。
