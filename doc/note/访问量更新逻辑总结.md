
## 访问量更新逻辑总结

### 一、访问量更新逻辑（实时更新）

#### 触发时机
用户访问书籍详情页时，前端调用 `addVisitCount(bookId)` 接口

#### 执行流程

**1. Lua 脚本原子操作** (`incrementVisitCount.lua`)

```lua
每次访问时执行：
1. ZINCRBY 更新排行榜 ZSet
   - Key: Cache::Novel::visit_rank
   - 操作: 分数 +1
   - 作用: 实时更新访问排行榜排名
   
2. HINCRBY 更新访问量缓冲 Hash
   - Key: Cache::Novel::book_visit_buffer
   - 操作: 值 +1（累计增量）
   - 作用: 累计增量访问量，用于批量同步到数据库
   
3. 如果书籍详情 Hash 存在，HINCRBY 更新详情缓存
   - Key: Cache::Novel::book_info:{bookId}
   - 字段: visitCount
   - 操作: 值 +1（增量更新）
   - 作用: 实时更新详情页显示的访问量（仅 Top 100 热门书籍）
```

**2. 降级策略**
- 如果 Redis 异常，发送 MQ 消息，异步补偿

#### 数据实时性

| 数据         | 位置                 | 更新方式  | 实时性 |
| ------------ | -------------------- | --------- | ------ |
| 排行榜分数   | Redis ZSet           | `ZINCRBY` | 实时   |
| 访问量缓冲   | Redis Hash           | `HINCRBY` | 实时   |
| 详情页访问量 | Redis Hash (Top 100) | `HINCRBY` | 实时   |

#### 特点
- 原子性：所有操作在一个 Lua 脚本中完成
- 高性能：单次网络往返，延迟约 2-3ms
- 实时性：排行榜和详情页访问量实时更新

---

### 二、访问量榜单更新逻辑（定时任务）

#### 定时任务：`BookRankCacheJob.refreshVisitRankCache()`

**执行频率**：每 30 分钟执行一次（兜底策略）

**执行逻辑**：

**1. 检查 ZSet 排行榜状态**

```
检查 ZSet 是否存在且数据正常
├─ 如果存在且正常 → 跳过重建，保留实时更新的数据 ✅
└─ 如果不存在或为空 → 从数据库重建（兜底）⚠️
```

**2. 查询数据库 Top 100 书籍**
- 按 `visit_count` 降序查询前 100 本
- 批量查询首章信息

**3. ZSet 排行榜重建（仅在兜底时）**
```
如果 ZSet 不存在或为空：
1. 优先从详情 Hash 读取实时访问量
2. 如果详情 Hash 不存在，使用数据库值
3. 删除旧的 ZSet 并重建
```

**4. 更新详情 Hash 缓存**
```
Pipeline 批量更新：
├─ 如果 Hash 已存在 → 排除 visitCount 字段，只更新其他字段
│  （避免覆盖实时更新的访问量）✅
└─ 如果 Hash 不存在 → 包含所有字段（包括 visitCount）初始化
```

#### 数据更新策略

| 数据                    | 更新策略 | 说明                       |
| ----------------------- | -------- | -------------------------- |
| ZSet 排行榜             | 兜底策略 | 存在则跳过，不存在才重建   |
| 详情 Hash 的 visitCount | 保留策略 | 已存在则不覆盖，保留实时值 |
| 详情 Hash 其他字段      | 定期更新 | 每 30 分钟更新一次         |

#### 特点
- 兜底机制：ZSet 异常时自动重建
- 数据保护：不覆盖实时更新的访问量
- 性能优化：降低执行频率，减少数据库压力

---

### 三、数据同步到数据库

#### 定时任务：`BookVisitSyncJob.syncVisitCount()`

**执行频率**：每 5 分钟执行一次

**执行流程**：
```
1. 原子重命名缓冲 Hash
   book_visit_buffer → book_visit_buffer:temp
   （防止处理过程中丢失新数据）
   
2. 读取临时 Hash 的所有数据
   （获取所有书籍的增量访问量）
   
3. 批量更新数据库
   UPDATE book_info SET visit_count = visit_count + ? WHERE id = ?
   
4. 删除临时 Key
```

**特点**：
- 批量处理：减少数据库压力
- 数据安全：使用原子重命名避免数据丢失
- 异步更新：不影响实时访问性能

---

### 四、完整数据流图

```
用户访问书籍详情页
    ↓
addVisitCount(bookId)
    ↓
Lua 脚本（原子操作，实时更新）
    ├─→ ZSet 排行榜 ✅ 实时
    ├─→ 缓冲 Hash ✅ 实时（累计增量）
    └─→ 详情 Hash 的 visitCount ✅ 实时（仅 Top 100）
         ↓
    详情页显示 ← 从详情 Hash 读取 ✅ 实时显示
         ↓
定时任务（每30分钟，兜底）
    ├─→ 检查 ZSet：存在则跳过，不存在则重建
    └─→ 更新详情 Hash：保留 visitCount，更新其他字段
         ↓
定时任务（每5分钟）
    └─→ 批量同步缓冲 Hash 到数据库
         ↓
    数据库 visit_count 更新 ⏰ 延迟最多5分钟
```

---

### 五、关键设计点

1. 实时性优先：用户可见的数据（排行榜、详情页访问量）实时更新
2. 性能优化：使用 Lua 脚本保证原子性，单次网络往返
3. 数据保护：定时任务不覆盖实时更新的数据
4. 容错机制：ZSet 异常时自动重建，保证服务可用性
5. 批量同步：数据库和 ES 批量更新，降低压力

---

### 六、总结

- 访问量更新：用户访问时实时更新，通过 Lua 脚本保证原子性
- 榜单更新：定时任务作为兜底，不覆盖实时数据，降低执行频率
- 数据同步：定时批量同步到数据库，保证数据持久化

该设计在实时性、性能和数据一致性之间取得平衡，适合高并发场景。
