### 1.2 书籍排行榜（接口瘦身 + Redis 读优化）

**压测条件（三轮相同）：** `Number of Threads: 20`，`Ramp-up: 5s`，持续 **15 分钟**。

---

#### 面试可讲：问题 → 方案 → 技术点 → 效果

**遇到的问题：** 排行榜类接口为每条记录返回过多字段（整段 HTML 简介、封面长 URL、不必要元数据等），响应体大，序列化与网络开销高，高并发下吞吐与延迟明显变差。

**解决方法：**  
1. **展示与传输对齐**：前端只展示简介预览时，服务端只返回约 40 字纯文本（`RankBookDescUtils.toRankPreview`，与榜单一致）；DB/缓存写入策略按需求可只持久化或只回传必要长度。  
2. **按场景拆接口**：排行榜页用「表格瘦 DTO」；首页侧栏用「排名+书名，仅 Top1 带封面与简介」；「最新更新」单独接口只要表格五列，避免 30 本书全部「详情级」载荷。  
3. **Redis 读路径优化**：`Cache::Novel::visit_rank`（ZSet）与 `book_info:{id}`（Hash）**存储结构不变**；读榜单由 **`HGETALL` 整 Hash** 改为 **`HMGET` 只取**表格/侧栏需要的 field，减少 Redis 网络与 JVM 反序列化压力。

**相关技术点：** Redis ZSet 排行 + Hash 详情；**Pipeline 批量 HMGET**；DTO 按端裁剪；首页推荐 `homeBookCache` 同样缩短简介以减小缓存与响应体积。

**预期效果：** 单次响应更小、错误率下降、**吞吐显著上升**、P95/P99 延迟下降；无需迁移 Redis key 或改写入结构，回滚成本低。

## 主页优化与对并发请求的理解与实践（面试表述）

### 一句话

主页采用 **「运营推荐区局部聚合 + 榜单/更新区并行小接口」**；在 **不增加无意义往返** 的前提下，用 **故障隔离与独立缓存** 换稳定性，并对 **响应体与 Redis 读路径** 做瘦身，支撑更高并发。

### 我做了哪些主页相关优化

1. **局部聚合（`home/books`）**  
   轮播、顶部栏、本周强推、热门、精品等 **同属运营配置 + `home_book` 关联书单** 的模块，用 **一个接口一次返回列表**，前端按 `type` 拆分渲染。避免为每一块单独打配置接口，减少无意义请求数，缓存（如 `homeBookCache`）也可 **整块失效、整块刷新**。

2. **非聚合模块：前端并行请求**  
   点击/新书/更新榜侧栏、「最新更新」表格、友链等 **数据来源、更新频率、技术路径不同**（Redis ZSet+Hash vs 纯 DB 等），使用 **多个接口、组件内 `onMounted` / `Promise.all` 并行拉取**。避免「整页一个大聚合接口」带来的：**最慢模块拖死整包 TTFB**、**单点失败导致整页空白**。

3. **传输与缓存瘦身（与榜单优化同思路）**  
   首页推荐位简介与榜单一致，只传 **纯文本预览（约 40 字）**，减小 JSON 与 Redis 中 `homeBookCache` 体积，降低序列化与带宽压力。

4. **榜单类接口按场景再拆 + Redis `HMGET`**（详见上文 1.2 压测）  
   排行榜页表格、首页侧栏、最新更新 **各用瘦 DTO**；点击榜读 Redis 时 **按需字段 `HMGET`**，不写新 key、不改 ZSet/Hash 写入结构。

### 对「并发请求」的理解与实践

| 维度 | 做法与取舍 |
|------|------------|
| **为何不全页聚合** | 模块异构、耗时不同；聚合 = **长尾延迟叠加**；失败域过大。并行小接口 = **故障隔离**，单块失败可降级为空或重试，不全站白屏。 |
| **为何仍做局部聚合** | 强相关、同数据源、同缓存生命周期的块（运营推荐多位）合并为 **一次请求**，减少 RTT 与前端状态分叉，属于 **有边界的聚合**。 |
| **代价与缓解** | 并行会增加 **浏览器同时请求数、网关 QPS**；在 **HTTP/2 多路复用** 下额外开销可控；后端通过 **字段裁剪、缓存、HMGET** 控制单请求成本。 |
| **与压测的关系** | 主页多请求是 **架构选择**；榜单压测证明 **瘦响应 + Redis 部分读** 对吞吐与延迟帮助很大；二者叠加才是「能并行、也扛得住并行」。 |

### 可背诵的短答（30 秒版）

「主页我按数据边界拆：**运营推荐用 `home/books` 做局部聚合**；**榜单和最新更新并行多个小接口**，避免一个大接口被最慢子调用拖死、也避免一处失败整页挂。配合 **简介只传预览、榜单 HMGET**，在并行前提下把单次成本和 Redis 压力压下去。」
