### 1.2 书籍排行榜：`/api/front/book/visit_rank`（接口瘦身 + Redis 读优化）

#### 压测条件（①～③ 可比三轮）

| 项目 | 值 |
|------|-----|
| 压测时长（每轮） | **≥20 分钟** |
| JMeter | `Number of Threads: 20`，`Ramp-up: 5s` |
| 入口 | `GET http://120.53.89.103:1024/api/front/book/visit_rank` |

#### 面试可讲：问题 → 方案 → 技术点 → 效果

**问题**：排行榜单条返回字段过多（整段 HTML 简介、长 URL、非必要元数据），响应体大，序列化与带宽成本高；Redis 侧对 `book_info` 使用 **`HGETALL`** 放大网络与反序列化。

**方案**：

1. **展示与传输对齐**：简介仅回传约 **40 字** 纯文本预览（`RankBookDescUtils.toRankPreview` 等与 UI 一致）。
2. **按场景拆接口**：榜单页表格、首页侧栏、「最新更新」等使用不同瘦 DTO，避免「30 本书全是详情级载荷」。
3. **Redis 读优化**：`Cache::Novel::visit_rank`（ZSet）+ `book_info:{id}`（Hash）**写入结构不变**；读榜单由 **`HGETALL`** 改为 **`Pipeline + HMGET`** 按需取 field。

**技术点**：ZSet 排行 + Hash 详情；Pipeline 批量 HMGET；DTO 按端裁剪；首页 `homeBookCache` 同步缩短简介。

**效果（数据见下表）**：吞吐提升、P95/P99 下降、错误率收敛；无需迁移 key，回滚成本低。

#### 三轮优化对比（20 线程、长时压测）

| 阶段 | 吞吐 (req/s) | 平均 RT (ms) | Median (ms) | P90 | P95 | P99 | Error % |
|------|----------------|--------------|-------------|-----|-----|-----|---------|
| ① 初始（厚响应） | ~76.5 | 256 | 200 | 463 | 587 | 1039 | 0.197 |
| ② 简介仅约 40 字 | ~133.6 | 148 | 109 | 271 | 369 | 559 | 0.031 |
| ③ ② + `HMGET` 替代 `HGETALL` | ~316.5 | 62 | 39 | 99 | 195 | 321 | 0.000 |

**结论**：相同并发下，吞吐约 **77 → 134 → 317 req/s（约 4.1 倍）**；平均 RT **256 → 148 → 62 ms**；瓶颈主要在 **响应体体积 + Redis 全量读 Hash**。`Received KB/s` 三轮约 **2060 KB/s** 量级时，吞吐提升主要来自**服务端处理变快**（单位时间完成更多请求）。

## 3. 主页优化与并发请求（面试表述）

### 一句话

主页采用 **「运营推荐区局部聚合 + 榜单/更新区并行小接口」**；在 **不增加无意义往返** 的前提下，用 **故障隔离与独立缓存** 换稳定性，并对 **响应体与 Redis 读路径** 做瘦身，支撑更高并发。

### 我做了哪些主页相关优化

1. **局部聚合（`home/books`）**  
   轮播、顶部栏、本周强推、热门、精品等同源运营配置模块，**一个接口一次返回**，前端按 `type` 渲染；缓存（如 `homeBookCache`）可整块失效与刷新。

2. **非聚合模块并行请求**  
   榜单侧栏、「最新更新」、友链等 **数据来源与更新频率不同**，多接口 + `onMounted` / `Promise.all` 并行，避免单聚合接口 **长尾拖死 TTFB**、单点失败整页空白。

3. **传输与缓存瘦身**  
   推荐位简介与榜单一致，仅 **~40 字** 预览，降低 JSON 与 Redis 体积。

4. **榜单 HMGET**  
   与 1.2 压测一致：读 `book_info` 按需 **HMGET**，不写新 key。

### 对「并发请求」的理解（简表）

| 维度 | 做法与取舍 |
|------|------------|
| 不全页聚合 | 模块异构、耗时不同；聚合 = 长尾叠加、失败域大。 |
| 局部聚合 | 强相关、同缓存生命周期模块合并，减少 RTT。 |
| 代价 | 并行增加浏览器与网关 QPS；靠 **HTTP/2**、**瘦 DTO、HMGET** 控制单请求成本。 |
| 与压测 | 榜单数据证明 **瘦响应 + HMGET** 可支撑更高并行度。 |

### 30 秒版
