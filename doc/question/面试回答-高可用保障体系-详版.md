# 高可用保障体系（面试详版）

> 本文档基于 **turtle-website** 仓库当前实现整理，结构为：**场景 → 技术选型 → 做法 → 效果 → 未来改进**。  
> 用途：面试时按「分层防护 + 超时链 + 业务语义」讲述，避免空泛背诵。

---

## 总览：分层与调用链

```
用户 / 前端
    → Nginx（若有）→ Spring Cloud Gateway（Sentinel 路由级 QPS + 全局/AI 超时）
          → lb:// 各微服务（Nacos 服务发现 + LoadBalancer 重试）
                → 业务层（DB HikariCP、Redis、RocketMQ）
                → 服务间 OpenFeign（全局/按服务超时 + Sentinel 集成 + 部分 Fallback）
```

**一句话**：入口用 **网关限流 + 超时** 挡洪峰；进程内用 **连接池/Redis/MQ 超时** 避免占满线程；跨服务用 **Feign 超时 + Sentinel + Fallback** 隔离故障；部分热点接口再用 **Redis Lua 业务限流** 精细控量。

---

## 1. 网关层：按路由 QPS 限流（全站入口保护）

| 维度 | 说明 |
|------|------|
| **场景** | 突发流量、爬虫、异常客户端集中打某一类 URL（如登录、搜索、作家后台）；若不限制，会放大到下游 JDBC/Redis，拖垮整个集群。 |
| **技术选型** | **Spring Cloud Alibaba Sentinel** + `spring-cloud-alibaba-sentinel-gateway`；规则资源维度为 **Gateway `routeId`**。 |
| **做法** | 在 `novel-gateway` 中通过 `GatewaySentinelConfiguration` 读取 `novel.gateway.sentinel.routes`，为各路由注册 `GatewayFlowRule`（`RESOURCE_MODE_ROUTE_ID`），例如：`novel-user-login: 30`、`novel-search-front: 200`、`novel-ai-front: 80` 等（单实例、每秒 QPS，具体数值以仓库 `application.yml` 为准）。登录单独拆路由且 **order 更高优先级**，便于单独限流。可选接 Sentinel Dashboard（`SENTINEL_DASHBOARD_ADDR`）。阻断时由 `BlockRequestHandler` 返回统一 JSON 提示。 |
| **效果** | 在**最外层**削峰：超出阈值的请求不会进入业务服务，保护线程与连接池；登录与 AI 等高风险/高成本路由可单独配额。 |
| **未来改进** | ① 规则从 **Nacos 动态下发**（代码里已预留「接 Nacos」的注释），避免改 YAML 发版；② 结合 **集群流控**（多网关实例时按总量限流）；③ 对真实客户端可叠加 **用户维度** 或 **API Key**（网关自定义 KeyResolver），与下文业务限流形成互补。 |

## 2. 网关层：全局与分路由 HTTP 超时（超时链最上游）

| 维度 | 说明 |
|------|------|
| **场景** | 下游已宕或极慢时，若网关无限等待，会占用连接与线程，造成**雪崩**；AI/生图类接口耗时可长达分钟级，与 60s 默认路由不一致。 |
| **技术选型** | Spring Cloud Gateway **全局** `httpclient.response-timeout` + 路由 **metadata** 覆盖。 |
| **做法** | 全局 `response-timeout: 60s`，`connect-timeout: 5s`；`novel-ai-front` 路由在 `metadata.response-timeout: 180000`（毫秒）单独放大。注释中与 **Feign**（默认 50s、`novel-ai-service` 175s）、**Hikari** 取连接 20s、**Redis** 命令 10s 对齐，形成「由外向内递减」的超时链，避免出现「下游已断、网关仍在等」。 |
| **效果** | 慢请求可 **fail-fast**，释放网关资源；AI 长链路有独立预算，减少误杀正常生图请求。 |
| **未来改进** | ① 按 **接口级** 更细粒度超时（需拆分路由或使用 Gateway Filter）；② 与 **SLO**（如 P99）挂钩定期调参；③ 对 SSE/WebSocket 若引入需单独策略（当前文档以 HTTP 为主）。 |

## 3. 负载均衡：实例级故障转移（调用侧轻量容错）

| 维度 | 说明 |
|------|------|
| **场景** | 多实例注册到 Nacos 时，某一实例不健康或重启，希望 **自动换实例** 而非把错误直接抛给用户。 |
| **技术选型** | Spring Cloud **LoadBalancer** + `spring.cloud.loadbalancer.retry.enabled: true`（见网关配置）。 |
| **做法** | 网关 `uri: lb://service-name`；在实例短暂不可用或网络抖动时，重试可落到健康实例（具体重试策略以 Spring Cloud 版本默认行为为准，生产可再显式配置）。 |
| **效果** | 提升 **可用性** 与请求成功率，尤其在滚动发布期间。 |
| **未来改进** | ① 明确 **幂等** 接口才重试 POST（避免重复写）；② 与 **熔断** 联动，避免对已知故障实例反复重试。 |

## 4. 业务层：Redis + Lua 令牌桶限流（热点昂贵接口）

| 维度 | 说明 |
|------|------|
| **场景** | **AI 封面生图** 成本高、易被刷；仅靠网关按路由 QPS 无法区分「同一用户短时间多次点击」。 |
| **技术选型** | **AOP** + **StringRedisTemplate** + **Lua 脚本**（Hash 存 `tok`/`ts`，`TIME` 驱动补充，原子 `HSET`），配置项 `novel.user.ratelimit.cover-image`（`CoverImageRateLimitProperties`）。 |
| **做法** | `CoverImageRateLimitAspect` 拦截带 `@CoverImageRateLimit` 的接口；桶 **容量** = `max-permits`，**补充速率** = `max-permits / window-seconds`（令牌/秒）；Key 形如 `ratelimit:{author:ai:cover}:tb:user:{userId}`（与历史固定窗口 STRING key 区分，避免 WRONGTYPE）。超限返回 `AI_IMAGE_GENERATION_BUSY`。可通过 `enabled` 关闭限流便于测试。 |
| **效果** | **用户维度** 保护 AI 资源与成本；相对固定窗口 **平滑窗口边界突发**；与网关 **路由级** 限流互补。 |
| **未来改进** | ① **滑动窗口** 与令牌桶对比选型文档化；② 多实例下「全集群统一配额」与 Sentinel 集群流控组合；③ 按 **作者 ID** 维度 Key 防多账号绕开。 |

## 5. 服务间调用：OpenFeign 超时 + Sentinel 集成

| 维度 | 说明 |
|------|------|
| **场景** | 微服务间 RPC 若无限等待，会占满 Tomcat/业务线程；需与网关超时 **协调**，并支持 **熔断降级**（与 Sentinel 配合）。 |
| **技术选型** | **OpenFeign** + `spring.cloud.openfeign.client.config`；`feign.sentinel.enabled: true`（`application-common.yml`）。 |
| **做法** | **默认**：`connect-timeout: 5000`，`read-timeout: 50000`；对 **`novel-ai-service`** 单独 `read-timeout: 175000`，与网关 AI 路由 180s、内部生图链路一致。注释强调：Hikari `connection-timeout` 应明显小于 Feign read-timeout，避免「等 Feign 时长期占着 DB 连接」。 |
| **效果** | 下游慢/挂时 **快速失败**；AI 长任务不被默认 50s 误杀。Sentinel 开启后，可在资源维度做流控/熔断（若已配置规则或 Dashboard）。 |
| **未来改进** | ① 对 **非 AI** 的慢接口再拆分 **按方法/契约级** 超时（需 `configuration` 或拆分 Feign）；② 在 Dashboard/Nacos 中落地 **错误比例熔断**、**慢调用比例** 等规则并写入面试案例数据。 |

## 6. Feign 降级：Fallback / FallbackFactory（故障时可控响应）

| 维度 | 说明 |
|------|------|
| **场景** | `novel-book-service`、`novel-ai-service`、`novel-user-service` 之一超时或不可用时，调用方不能无限阻塞或直接 500 穿透到前端；需要 **可预期的错误码或空数据**。 |
| **技术选型** | Feign **`fallback`** / **`fallbackFactory`**（带异常原因）。 |
| **做法** | - **`AiFeign`**：`AiFeignFallback` 对各方法返回 `RestResp.fail` 与业务错误码（如 `AI_AUDIT_SERVICE_ERROR`、`AI_COVER_TEXT_SERVICE_ERROR`）。  <br> - **`BookFeign`**：`BookFeignFallbackFactory` 记录 `cause` 日志，按方法返回空列表或失败响应。  <br> - **`UserFeign`**：`listUserInfoByIds` 失败时返回空列表，避免列表页全挂；`sendMessage` 失败返回系统错误。 |
| **效果** | **故障隔离**：依赖挂掉时主流程可 **优雅失败**；`FallbackFactory` 便于排障。 |
| **未来改进** | ① **`AuthorFeign`**（积分扣减/回滚）、**`SseNotificationFeign`**（推送）当前 **未配置 Fallback**，熔断/异常时行为需与业务确认（见下节）；② 对「必须成功」的写操作，Fallback 仅返回失败不够，需 **补偿任务 / 对账**（见 MQ 与业务设计）。 |

## 7. 数据层与中间件：连接池与 Redis/MQ 超时（避免资源耗尽）

| 维度 | 说明 |
|------|------|
| **场景** | 高并发下，若连接池过大或等待过久，会堆积请求；Redis/MQ 慢会拖住业务线程。 |
| **技术选型** | **HikariCP**；**Lettuce** Redis；**RocketMQ** Producer 超时。 |
| **做法** | `application-common.yml`：Hikari `maximum-pool-size`、`connection-timeout`（注释中与 Feign 协调）；Redis `connect-timeout: 3s`、`timeout: 10s`（注释：应远小于网关 60s）；RocketMQ `send-message-timeout: 3000`。 |
| **效果** | 避免 **线程长时间阻塞在 IO**；消息发送失败尽快感知。 |
| **未来改进** | ① 各服务按 **QPS 与连接公式** 单独调池大小（已有压测文档可参考 `doc/note/压测与内存分配、数据库连接池分配调整.md`）；② MQ 消费端 **幂等 + 重试 + 死信** 完整故事可单独一节（若代码已部分实现，可在面试中点名类名）。 |

## 8. 异步与一致性：事务提交后再发 MQ（避免「消息已发、事务回滚」）

| 维度 | 说明 |
|------|------|
| **场景** | 书籍审核请求等需在 **DB 事务成功提交后** 再通知下游，否则会出现数据不一致。 |
| **技术选型** | Spring **`TransactionSynchronizationManager.registerSynchronization(afterCommit)`** + **RocketMQTemplate**。 |
| **做法** | 如 `BookSubmitListener.sendAuditRequest`：在 `afterCommit` 中 `convertAndSend`，避免事务未提交就发 MQ。 |
| **效果** | 提升 **最终一致性** 概率，减少脏消息。 |
| **未来改进** | ① 与 **事务消息**（若升级 RocketMQ 事务消息能力）或 **Outbox 表** 对比，说明极端故障下的选择；② 监控 **发送失败** 与重试队列。 |

## 9. 业务规则：减少无意义下游负载（审书触发条件）

| 维度 | 说明 |
|------|------|
| **场景** | 前端 **全量提交** `updateBook`（如仅换封面）时，若消费者仅凭「字段非空」就触发 AI 审书，会造成 **重复审核、浪费算力**。 |
| **技术选型** | 业务侧 **幂等/等价判定**（字符串 trim 后比较），不依赖 Redis 预载。 |
| **做法** | `BookSubmitListener.handleBookUpdate`：在已有 `selectById` 结果上比较书名/简介是否与库中 **真实变化**，仅此时 `needAudit` 与发审书 MQ；仅改封面/分类等不触发。 |
| **效果** | 降低 **AI 与 MQ 压力**，审核队列更干净。 |
| **未来改进** | ① 前端 **按需提交变更字段**（减少带宽与歧义）；② 规范化 **Unicode/全角空格** 若业务需要可再收紧。 |

## 10. 可观测性：链路追踪与指标（高可用的「眼睛」）

| 维度 | 说明 |
|------|------|
| **场景** | 限流/超时/降级发生时，需快速定位 **哪一跳慢、是否触发 fallback、是否 Sentinel 阻断**。 |
| **技术选型** | 文档与代码中涉及 **SkyWalking**（如 AI Pipeline Listener）、**Micrometer**（如 `MicrometerPipelineListener` 对 step outcome 打点）；网关/服务 **Actuator** 暴露端点。 |
| **做法** | 在 AI 审核管道中对 `success` / `short_circuit` / `error` 等打标签或指标；结合 trace 查 Feign 与 LLM 耗时。 |
| **效果** | 面试可讲：**不是只堆组件，而是能把故障复盘到「网关 429 → 下游超时 → Fallback」闭环**。 |
| **未来改进** | ① **统一仪表盘**：Sentinel 阻断次数、Feign 错误率、网关 5xx 同屏；② **告警** 与 on-call 流程；③ **SLO/错误预算** 量化。 |

## 面试表达模板（2～3 分钟）

1. **分层**：网关 Sentinel + 超时是第一道；业务 Redis 限流保护贵接口；Feign 超时与 Sentinel 是第二道；Fallback 是最后一道用户体验与错误码。  
2. **超时链**：从外到内 **递减**，避免外层挂等、内层占连接——结合你们 `application-common.yml` 与网关注释原话。  
3. **诚实缺口**：`AuthorFeign` / `SseNotificationFeign` 无 Fallback，生产上会补 FallbackFactory 或明确「失败重试/补偿」策略。  
4. **业务高可用**：不只基础设施，还有 **MQ 事务提交后发送**、**审书条件收敛** 等减少无效负载。  
5. **未来**：动态规则、集群流控、幂等与重试策略、SLO 驱动调参。
