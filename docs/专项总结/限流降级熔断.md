# turtle-website：限流、降级、熔断

本文档基于 **当前仓库实现** 归纳：说明各层在做什么、关键配置与代码位置，并区分 **「流量控制」**、**「故障容错/降级」** 与 **「Sentinel 熔断规则（代码侧）」** 在本项目中的落地情况。

---

## 一、总览：分层与配合关系

| 层次 | 手段 | 作用 |
|------|------|------|
| **网关** | Spring Cloud Alibaba **Sentinel**（按 `routeId` 的 QPS） | 入口削峰，超限请求不进入下游 JVM |
| **网关** | `httpclient` 连接/响应超时；AI 路由单独拉长 `response-timeout` | 避免网关线程长时间挂死在慢下游 |
| **跨服务调用** | OpenFeign 超时（全局 / `novel-ai-service` 单独）+ `spring.cloud.loadbalancer.retry` | 控制单次 RPC 时长；失败可重试一次（网关已开 retry） |
| **跨服务调用** | `feign.sentinel.enabled: true` + Feign **Fallback / FallbackFactory** | Sentinel 可接管 Feign 资源名；调用异常时走降级实现 |
| **业务热点** | `novel-user`：**Redis 令牌桶**（Lua）限 AI 封面生图 | 用户维度控量，多实例共享桶状态 |

**说明**：仓库 Java 代码里**仅见**网关侧的 `GatewayFlowRule`（流控）。未发现通过代码注册的 `DegradeRule`（熔断）。跨服务容错目前主要体现在 **超时链 +（可选）LB 重试 + Feign 降级类**；若需 Sentinel 标准熔断/集群流控，需依赖 **Sentinel Dashboard** 或 **Nacos 规则推送**（网关配置类注释中已提及后续方向）。

## 二、限流

### 1）网关：按路由 `routeId` 的 QPS（Sentinel）

- **实现**：`novel-gateway` 中 `GatewaySentinelConfiguration` 在启动时读取 `novel.gateway.sentinel.routes`，为每个键（与 `spring.cloud.gateway.routes[].id` 一致）注册 `GatewayFlowRule`，资源模式为 **`RESOURCE_MODE_ROUTE_ID`**，档位为 **QPS**，统计窗口 **1 秒**。
- **配置位置**：`novel-gateway/src/main/resources/application.yml` → `novel.gateway.sentinel.routes`。
- **默认阈值（以仓库当前 YAML 为准，单实例、每秒）**：如 `novel-user-login: 30`、`novel-search-front: 200`、`novel-ai-front: 80`、`novel-book-front: 100`、`novel-user-front: 150`、`novel-author: 80`、`novel-resource-front: 120`。
- **阻断响应**：`BlockRequestHandler` 返回 **HTTP 429**，Body 为统一 JSON（`code: 42901`，提示请求过于频繁）。
- **开关**：`novel.gateway.sentinel.enabled: false` 时不加载上述规则（依赖仍保留，便于接 Nacos）。
- **可选运维**：`spring.cloud.sentinel` 可与环境变量 **`SENTINEL_DASHBOARD_ADDR`** 配合接 Dashboard（见网关 `application.yml` 注释）。

### 2）用户服务：AI 封面生图 — Redis 令牌桶（业务限流）

- **实现**：`CoverImageRateLimitAspect` 拦截 `@CoverImageRateLimit`，使用 **Redis Hash + Lua** 实现令牌桶；按 **用户 ID** 分 key，多实例共享。
- **配置**：`novel-user-service` 的 `application.yml` → `novel.user.ratelimit.cover-image`（如 `enabled`、`max-permits`、`window-seconds`）。
- **超限**：返回业务错误码 **`AI_IMAGE_GENERATION_BUSY`**（与 AI 侧 C4002 类文案对齐）；Redis 脚本异常等场景会 **降级放行**（打日志），避免限流逻辑拖垮主流程。

### 3）其他

- 仓库内**未发现**业务代码中大量使用 `@SentinelResource` 做方法级限流；入口限流以 **网关** 为主，热点以 **Redis 业务限流** 为主。

## 三、降级（语义上的「弱化响应、保主链路」）

### 1）Feign：`fallback` / `FallbackFactory`

全局开启 **`feign.sentinel.enabled: true`**（见 `novel-config` 的 `application-common.yml`），Feign 与 Sentinel 集成后，资源名一般与方法签名对应；**具体熔断/限流规则**仍须在 Sentinel 侧配置（代码库未写死 `DegradeRule`）。

各客户端降级行为示例：

- **`AiFeign`**：`AiFeignFallback` 在异常时返回 `RestResp.fail(...)`（如 `AI_AUDIT_SERVICE_ERROR`、`SYSTEM_ERROR` 等），避免上游 NPE 或未定义行为。
- **`BookFeign`**：`BookFeignFallbackFactory` 可按 `cause` 打日志；例如书架列表在异常时用占位书名/描述提示「系统维护中…」，列表接口多返回空分页或空列表。
- **`UserFeign`**：`UserFeignFallback` 在异常时批量用户查询返回空列表，`sendMessage` 返回失败等。

以上属于典型的 **容错降级**：下游不可用或超时时返回约定结构，不拖死调用方线程（配合 Feign **read-timeout**）。

### 2）领域内的「降级」实现（非 Feign）

- **搜索**：向量或混合检索失败时 **降级为纯文本搜索**（见 `novel-search` 检索实现中的 catch 日志与分支）。
- **AI 模块**：判例向量检索等失败时 **降级为空召回**；润色等场景在字段缺失时 **降级为原文 + 说明**；`RetryTransientAiAdvisor` 仅对 **`TransientAiException`**（限流、超时、5xx 等）做指数退避重试。
- **其他**：缓存解析失败 **降级查库**、MQ 只发异步等（属数据路径降级，与「熔断器」概念不同）。

## 四、熔断与「故障隔离」在本项目中的含义

### 1）代码侧 Sentinel **熔断规则**

- **未发现**在 Java 中注册 **`DegradeRule`**（熔断）或与网关流控并列的 **Feign 方法级** Sentinel 规则。
- 因此：**不能**仅凭仓库代码断言「已配置 Sentinel 慢调用比例熔断 / 异常比例熔断」；若线上有，通常来自 **Dashboard / Nacos 推送**，需以实际运行时配置为准。

### 2）实际起到的隔离作用（可面试表述）

- **超时**：Feign `default` 与 `novel-ai-service` 分档；网关全局 **60s**、AI 路由 **180s**，与下游及连接池等待时间形成链条，避免无限阻塞。
- **重试**：网关 **LoadBalancer retry** 开启，用于瞬时网络抖动；与业务幂等需自行保证的场景要结合设计（见 `novel-gateway` `application.yml`）。
- **Fallback**：调用失败快速返回业务可识别的失败或占位数据，避免级联故障表现为大面积 500。
- **AI Pipeline `SHORT_CIRCUIT`**：审核流水线中某步已产出结果或校验不通过时 **提前结束步骤链**，这是 **流程控制语义**，不是分布式熔断器。
