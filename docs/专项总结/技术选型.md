# 技术选型

## 面试问题：项目体量不大，为何拆分微服务？

---

我在做这个项目之前，也纠结过要不要直接做成一个单体应用。毕竟用户量不大，拆成微服务反而增加复杂度。

但是**技术选型不能只看眼前，要看项目的演进方向**。

我这个小说网站虽然现在体量小，但它有几个业务模块的差异非常大：

- **搜索模块**：用的是 Elasticsearch，走的是 IO 密集型路线，读多写少，天然适合独立扩展
- **AI 模块**：要对接通义千问的 API，依赖很重，而且上游 API 还有限流问题，这个必须独立管理
- **用户模块**：主要是 Token 验签、密码加密这类 CPU 密集的计算

这三个模块的功能特点完全不一样，如果做成单体，部署的时候只能取个折中——既不够 IO 优化，也不够 CPU 优化，反而浪费。

而且说实话，我也把它当作学习 Spring Cloud 的实践项目。因为**在体量小的时候拆微服务，代价是最低的**——我可以在不影响业务的前提下，尽情踩坑。等哪天真的做大了起来再拆，成本就高多了。

---

### 适度拆分，不过度设计

当然我也注意没有过度拆分。

如果按"用户表""书籍表"一个个拆反而增加复杂度，我的拆分原则是**按业务边界和资源需求差异**进行拆分：

- `novel-user` - 用户相关
- `novel-book` - 书籍相关
- `novel-search` - 搜索（ES + RocketMQ）
- `novel-ai` - AI 能力（对接大模型）

然后公共的代码抽到 `novel-common` 和 `novel-config` 里复用，避免重复建设。

---

### 拆分后实际带来的好处

举两个我实际遇到的场景：

**场景一：重构某模块不需要全量发布**

之前我重构了登录模块的逻辑，换了一套 Token 刷新机制。如果是单体应用，我得把整个系统重新打包部署一次。

现在是微服务，我只需要在 Gateway 路由不变的情况下，重启 novel-user-service 就行。novel-book-service、novel-search 完全无感知，出了问题和回滚都只需要盯着用户服务这一个点。

**场景二：资源隔离，互不干扰**

有一次搜索服务在做全量索引，CPU 飙得很高。如果是单体应用，整个系统都会被拖慢。但现在搜索服务独立部署，只是它自己的响应变慢了，其他服务完全不受影响。

---

### 总结

我觉得拆分微服务不只是为了"赶时髦"，而是**这个项目的业务特点确实适合拆，而且早拆早好**。等真的做大了，用户量上来了，再去拆，成本和风险都会高很多。

---

## 面试加分项：主动提及的代价和应对

如果面试官顺着追问"那微服务有什么代价？"，可以这样接：

> "代价肯定是有的。首先是运维复杂度上来了，我需要用 Docker + docker-compose 来管理，还要做 CI/CD 自动部署。其次是分布式事务的问题，跨服务的数据一致性需要额外处理，我目前用的是 RocketMQ 做最终一致性，核心场景用消息+补偿机制兜底。"
>
> "另外调用链路变长了，排查问题更复杂，所以我引入了 SkyWalking 做链路追踪。"

---

## 追问：你们的服务是怎么治理的？

> "用 Nacos 做注册中心和配置中心，Gateway 做路由分发，Gateway 上配了限流和熔断。服务间调用用 OpenFeign，走的是 LoadBalancer。"

如果能画个简单的架构图会更好：
```
用户请求 → Gateway → Nacos (发现/配置)
                    ├── novel-user (用户服务)
                    ├── novel-book (书籍服务)
                    ├── novel-search (ES搜索)
                    └── novel-ai (AI服务)
```

---

## 追问：跨服务数据一致性怎么做的？

> "其实大多数业务场景不需要强一致性，我们用的是 RocketMQ 做最终一致性。比如用户购买书籍，支付成功后，发一条消息给书籍服务更新会员状态，消息消费失败有重试机制，最终会达到一致。"
>
> "如果真的有强一致性需求，比如下单和库存必须同时成功，可以用 Seata 的 AT 或 TCC 模式。不过目前我的场景还没到这个程度，用消息队列+定时任务补偿就够了。"

## 为什么要用 RocketMQ？和 Kafka、RabbitMQ 怎么选的？

---

### 先说我的实际业务场景

我项目里 RocketMQ 主要用在两类场景：

**1. 异步解耦 - 章节审核流程**

作者提交章节 → 发送 MQ 消息 → AI 服务消费 → 做内容审核 → 回调通知

在这个流程里，AI 审核是耗时操作，如果走同步调用，用户要等很久才能知道结果。用 MQ 异步处理后，用户提交了就去干别的事，系统后台慢慢审核，完了再通知。

**2. 事务一致性 - 书籍更新通知**

章节发布成功 → 必须同时更新书籍信息 + 字数统计 + 发送通知给订阅用户

这里有一个细节：消息必须等数据库事务提交后再发，不能事务还没提交就发出去，否则数据不一致。我的代码里用的是 `TransactionSynchronizationManager` 注册的 `afterCommit` 回调，确保事务提交成功才发 MQ。

---

### 为什么选 RocketMQ 而不是 Kafka 或 RabbitMQ？

**Kafka 的取舍**

Kafka 的优势是**高吞吐、低成本**，适合日志收集、流量削峰这种"丢几条无所谓"的场景。

但我的需求不太一样：
- 审核消息、通知消息**不能丢**，得保证可靠投递
- Kafka 默认是"at least once"，要实现 exactly-once 代价很高
- 而且 Kafka 不支持**消息延迟投递**（RocketMQ 有 delay message）

**RabbitMQ 的取舍**

RabbitMQ 功能完善，支持多种消息模式，延迟队列也有。

但它有两个问题：
- **主从复制不如 RocketMQ 稳**，在金融级场景用得少
- **集群部署复杂度高**，运维成本大
- 单机吞吐量不如 RocketMQ

**RocketMQ 的优势**

| 特性 | RocketMQ | Kafka | RabbitMQ |
|-----|---------|-------|---------|
| 吞吐量 | 高 | 极高 | 中 |
| 消息可靠性 | 好（事务消息） | 一般 | 好 |
| 延迟消息 | 支持 | 不支持 | 支持 |
| 事务消息 | 原生支持 | 需自实现 | 需插件 |
| 运维复杂度 | 中 | 中 | 高 |

最终选 RocketMQ，主要看中两点：

**1. 事务消息原生支持**

这个对我很重要。比如"章节发布成功"这个操作，数据库更新和 MQ 投递必须是原子的——要么都成功，要么都失败。RocketMQ 的事务消息机制可以做到，Kafka 要自己实现两阶段提交，太麻烦。

**2. 延迟消息场景**

比如一本书被举报了，我要"1小时后自动取消屏蔽"。这个用 RocketMQ 的延迟消息很简单，Kafka 得搭额外的定时任务。

---

### 总结

> "其实没有最好的消息队列，只有最合适的。我的场景需要可靠的事务消息 + 延迟投递能力，RocketMQ 最合适。如果是大数据日志那种场景，Kafka 明显更优。"

## 为什么要用 Elasticsearch？

---

### 先说我的实际业务场景

我的小说网站里，搜索是核心功能之一。用户可以：
- 搜书名、作者名
- 按分类、字数、更新时间筛选
- **更重要的是：用户可以搜情节**

"搜情节"这个需求很有意思——比如用户记得小说里有一段"主角穿越到古代当王爷"，但说不清书名，只记得情节。那传统的关键词匹配就不够用了，需要**语义搜索**能力。

---

### 我的 ES 使用方案：Hybrid Search 混合搜索

这是我觉得比较有技术含量的部分。

**架构是这样的：**

```
用户输入 → 通义千问生成向量 → ES KNN 向量检索
                      ↓
              同时走 BM25 文本检索
                      ↓
              Java 端手动融合结果（权重 6:4）
```

**具体实现：**

1. **书籍入库时**：用通义千问的 embedding 接口，把"书名 + 作者 + 简介"生成一个向量，存入 ES 的 `embedding` 字段

2. **搜索时**：
   - 短关键词（如"斗破苍穹"）→ 走纯文本 BM25，精准匹配书名
   - 长句/情节描述（如"主角穿越到古代当王爷"）→ 走向量检索 KNN，找语义相似的结果
   - 智能模式：关键词超过6个字或包含空格 → 自动切换向量检索

3. **结果融合**：KNN 和 BM25 各召回一批结果，在 Java 端按排名加权合并，而不是用 ES 内置的 RRF

这样做的好处是：两种检索方式互补——文本检索精确，向量检索语义理解强。

---

### 为什么要用 ES 而不是 MySQL 全文索引？

| 对比维度 | MySQL 全文索引 | Elasticsearch |
|---------|---------------|---------------|
| 中文分词 | 效果一般 | IK 分词器效果好 |
| 语义搜索 | 不支持 | 支持向量检索 |
| 性能 | 数据量大了很慢 | 海量数据搜索快 |
| 分布式 | 不支持 | 原生支持 |
| 搜索相关性 | 一般 | 可调优 |

而且 ES 还有个好处是**搜索和数据库解耦**——搜索索引崩了不影响主业务，数据库压力也小很多。

---

### ES 数据同步方案

书籍数据在 MySQL，但搜索索引在 ES，这里有一个同步问题。

我的方案是**基于 RocketMQ 的最终一致性**：

```
书籍发布/审核通过 → 发送 MQ 消息 → Search 服务消费
                                        ↓
                              调用 Book 服务获取最新数据
                                        ↓
                              生成向量 → 更新 ES 索引
```

这里有个细节：如果某本书审核不通过，ES 里本来就搜不到，因为同步的时候 `getEsBookById` 返回 null，会从 ES 删除这本书。

---

### 追问：ES 集群部署了吗？数据量大吗？

> "目前是单节点部署，因为体量还不大。但 ES 本身就是支持集群的，真到了大数据量，可以水平扩展分片。不过说实话，如果只是几百万本书这种量级，单节点完全够用了。"

---

### 追问：向量检索的向量是怎么生成的？

> "用的是通义千问的 embedding 接口，书名+作者+简介拼成一段文本，调用接口生成一个 1024 维的向量。生成向量的时候有 500ms 延迟，避免触发上游限流。"

---

### 总结

> "选 ES 主要是因为我的搜索场景比较复杂——既要有关键词精准匹配，又要有语义搜索能力。MySQL 做不到，ES 的 Hybrid Search 刚好满足。而且 ES 本身是专业的搜索引擎，在搜索体验优化（高亮、分词器、相关性调优）上比自己实现强太多了。"

---

## 为什么要用 Redis？

---

### 先说我的实际业务场景

我在项目里用 Redis 主要解决这几类问题：

**1. 热数据缓存 - 减少数据库压力**

比如小说章节内容、书籍信息、用户信息，这些都是读多写少的场景。直接查数据库太浪费，Redis 缓存起来，响应快得多。

而且我用的是**多级缓存**策略——本地 Caffeine + 远程 Redis。Caffeine 扛本地热点，Redis 扛分布式共享。

**2. 分布式 Session / Token 黑名单**

微服务架构下，用户登录后的 Token 存在 Gateway 做校验。但用户登出的时候，这个 Token 要失效。

我的做法是：登出时把 Token 的 MD5 哈希存入 Redis 黑名单，过期时间和 Token 本身过期时间一致。每次请求进来 Gateway 先查黑名单，在里面就直接拒绝。

**3. 排行榜/计数 - 用 ZSet 做书籍榜单**

点击榜、新书榜、更新榜这些需要排序的场景，用 Redis ZSet 实现特别合适。每次书籍被访问，在 ZSet 里 `ZINCRBY` 就行，天然支持按分数排序取 Top N，比数据库排序快几个量级。

**4. 流量缓冲 - 批量写数据库**

这个我觉得是做得比较巧的一个点。

小说网站的访问量很大，如果每次访问都直接更新数据库，数据库扛不住。

我的做法是：访问先落 Redis Hash，每本书的点击数先累加到 Redis，每 2 分钟定时任务批量同步到数据库。中间加了重试机制，同步失败了会保留数据下次重试，避免数据丢失。

---

### 为什么选 Redis 而不是本地缓存（如 Caffeine）？

本地缓存的问题是：**不能跨进程共享**。

我的服务是部署多个实例的（Docker 多容器），如果用纯本地缓存，用户访问 A 实例产生的缓存，B 实例看不到，缓存命中率很低。

Redis 作为分布式缓存，所有实例都读同一份数据，一致性好维护。

而且 Redis 还有额外好处：
- 支持过期时间，缓存自动失效
- 支持多种数据结构（String、Hash、ZSet、List）
- 支持 Pub/Sub，可以做实时通知

---

### 用的是 Redisson，它解决了什么问题？

一开始我用的是原生的 `StringRedisTemplate`，但后来换成了 Redisson，因为有些场景它更方便：

| 场景 | 原生 Redis | Redisson |
|-----|-----------|---------|
| 分布式锁 | 自己实现 | `RLock` 开箱即用 |
| 限流 | 攒 | `RateLimiter` |
| 原子计数 | `increment` | 更丰富的 API |
| 布隆过滤器 | 需装插件 | 内置 `RBloomFilter` |

不过说实话，如果项目里没有特别复杂的分布式协调需求，原生 `StringRedisTemplate` 也够用。Redisson 更适合那种"需要分布式锁但又不想自己造轮子"的场景。

---

### 追问：缓存和数据库一致性怎么保证的？

我实际用的是几种策略结合：

**1. Cache Aside（旁路缓存）**

读的时候先读缓存，缓存没有再读数据库，然后把数据写入缓存。写的时候先写数据库，再删缓存（不是更新缓存）。

**2. 延迟双删**

如果缓存删除失败了，会导致脏数据。我的做法是：删缓存 → 写数据库 → 延迟再删一次（几百毫秒后）。这样即使第一次失败，第二次也能兜住。

**3. 缓存过期 + 定时任务补偿**

最简单粗暴的方式：给缓存设一个过期时间，就算不一致，过期了就自动从数据库拉最新的。配合主动更新，基本没问题。

---

### 总结

> "Redis 在我项目里主要解决三类问题：缓存热点数据减少 DB 压力、分布式 Session 和 Token 黑名单、排行计算和流量缓冲。选它而不是纯本地缓存，是因为微服务多实例部署必须用分布式缓存保证一致性。而且 Redis 的数据结构丰富，ZSet 做排行、Hash 做计数器，都是很自然的选择。"
