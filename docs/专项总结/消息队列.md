# turtle-website RocketMQ 深度剖析

## 一、为什么用消息队列？业务痛点出发

在回答技术选型前，先要说清楚**没有 MQ 会怎样**：

### 痛点1：写操作链路太长，接口响应慢

作者发布一本书，如果同步处理：
```
保存书籍 → 调用AI大模型审核(秒级) → 更新审核状态 → 更新ES → 通知读者
```
整条链路用户要等几秒，大模型调用尤其不稳定。**用户体验极差**。

### 痛点2：服务间强耦合，一挂全挂

`novel-book-service` 如果同步调用 `novel-ai-service`，AI 服务抖动或重启时，书籍发布接口会直接报错。书籍服务不应该对 AI 服务有强依赖。

### 痛点3：流量洪峰打垮下游

章节发布触发的读者通知，可能涉及数万人，如果同步写数万条消息记录，数据库会被压垮。

**MQ 的核心价值：异步解耦 + 削峰填谷 + 保障最终一致性**。

## 二、为什么选 RocketMQ？

这是面试必问点，需要横向对比：

| 特性 | RocketMQ | Kafka | RabbitMQ |
|---|---|---|---|
| 消息顺序性 | 支持分区顺序 | 支持分区顺序 | 有限支持 |
| 消息延迟 | **原生支持延迟消息** | 不支持 | 插件支持 |
| 事务消息 | **原生支持** | 不支持 | 不支持 |
| 消费失败重试 | **自动重试16次+死信队列** | 需自行实现 | 支持 |
| 消息回溯 | 支持按时间回溯 | 支持 | 不支持 |
| Java生态集成 | **阿里系，与Spring生态深度集成** | 好 | 好 |
| 吞吐量 | 十万级/s | 百万级/s | 万级/s |

**选 RocketMQ 的核心理由：**
1. **顺序消息需求**：积分扣减必须有序（先扣免费积分，再扣付费积分），RocketMQ 天然支持
2. **可靠消费重试**：积分持久化失败必须重试，RocketMQ 自动重试 16 次 + 死信队列
3. **阿里系 Java 项目标配**：`rocketmq-spring-boot-starter` 集成极简，`@RocketMQMessageListener` 注解一行配置完成消费者
4. **业务量级适配**：小说平台 QPS 在万级以内，RocketMQ 的十万级吞吐完全够用，Kafka 的百万级吞吐是过度设计

## 三、Topic/Tag 设计哲学

项目共 **5 个 Topic，8 个 Tag**，来自统一常量类 `AmqpConsts.java`：

```14:60:novel-core/novel-common/src/main/java/com/novel/common/constant/AmqpConsts.java
public class AmqpConsts {
    public static class BookChangeMq {
        public static final String TOPIC = "topic-book-change";
        public static final String TAG_UPDATE = "update";
        public static final String TAG_CHAPTER_UPDATE = "chapter_update";
        // ...
    }
    public static class BookAuditRequestMq {
        public static final String TOPIC = "topic-book-audit-request";
        public static final String TAG_AUDIT_BOOK_REQUEST = "audit_book_request";
        public static final String TAG_AUDIT_CHAPTER_REQUEST = "audit_chapter_request";
        // ...
    }
    // ...
}
```

**为什么按 Topic 维度而不是全用一个 Topic？**

RocketMQ 的 Topic 是**物理隔离**的（不同 MessageQueue），Tag 是**逻辑过滤**的（同一队列内 Broker 过滤）。

- `topic-book-change`：书籍变更事件，ES服务和用户通知服务都关心，不同消费者组独立消费，互不影响
- `topic-book-audit-request` vs `topic-book-audit-result`：审核请求和审核结果分离成两个 Topic，AI 服务和书籍服务各自的读写职责清晰，体现了**请求-响应异步化**的设计模式
- `topic-author-points-consume`：积分操作独立 Topic，`deduct` 和 `rollback` 两个 Tag 在同一 Topic 下，因为它们共享同一类业务数据（积分），且消费顺序和路由逻辑相似

## 四、核心业务场景深度剖析

### 场景1：书籍发布 → AI 审核全链路（MQ 驱动的异步流水线）

```
作者点击"发布"
    │
    ▼  [同步，快速返回]
BookAuthorServiceImpl.saveBook()
    │  发送 MQ（不等结果）
    ▼
[topic-book-submit : submit]
    │
    ▼  [异步消费，独立事务]
BookSubmitListener
    ├── 书籍写入DB (auditStatus=0 待审核)
    └── 事务提交后 afterCommit() 再发送 ──→ [topic-book-audit-request : audit_book_request]
                                                        │
                                                        ▼
                                            BookAuditRequestListener (AI服务)
                                                调用 Qwen3-max 大模型
                                                        │
                                                        ▼
                                            [topic-book-audit-result : audit_book_result]
                                                        │
                                                        ▼
                                            BookAuditResultListener
                                            ├── 置信度≥0.8且通过 → 上架+清缓存+ES同步+通知作者
                                            ├── 置信度<0.8但通过 → 标记待人工审核
                                            └── 不通过 → 写拒绝原因，通知作者
```

**关键设计：`afterCommit()` 回调，事务提交后再发 MQ**

```206:225:novel-book/novel-book-service/src/main/java/com/novel/book/mq/BookSubmitListener.java
TransactionSynchronizationManager.registerSynchronization(new TransactionSynchronization() {
    @Override
    public void afterCommit() {
        rocketMQTemplate.convertAndSend(destination, auditRequest);
    }
});
```

**为什么不在事务内发 MQ？** 这是一个经典的分布式一致性问题：
- 如果在事务内发 MQ，MQ 发送成功但事务回滚 → AI 服务收到审核请求，但书籍不存在 → **数据幻读**
- 如果先发 MQ 再提交事务，事务提交失败 → AI 服务已开始审核不存在的书籍 → **资源浪费+脏数据**
- 用 `afterCommit()` 钩子，**确保 DB 落地后才发消息**，即使 MQ 发送失败，也可以通过定时任务补偿重发

**为什么用两个 Topic（请求/结果）而不是一个？**

AI 审核是一个耗时操作（调用大模型），如果用同一 Topic 同步等待，消费线程会被阻塞。两个 Topic 实现了**完全异步的请求-响应模式**，AI 服务消费请求、产出结果，书籍服务消费结果、更新状态，双方完全解耦，互不阻塞。

### 场景2：作者积分消费（Redis 先行 + MQ 异步落库）

这是 **Write-Behind（异步写回）** 模式在积分场景的应用，理解这个模式是面试加分项。

```
用户购买AI功能（调用扣积分）
    │
    ▼  [同步，快速返回]
AuthorServiceImpl.deductPoints()
    ├── 1. Redis SETNX 幂等锁（24h，防重复扣）
    ├── 2. Redis DECR 原子扣积分（先免费积分，不够再扣付费积分）
    └── 3. 发送 [topic-author-points-consume : deduct]（异步落库）
                │
                ▼
        AuthorPointsConsumeListener
        ├── 从 Redis 读最新积分值
        ├── 写入 DB author_info（以 Redis 为准）
        ├── 写积分日志（幂等键：idempotentKey_FREE / _PAID）
        └── 失败则 throw e → RocketMQ 自动重试

AI 调用失败时（积分回滚）：
AuthorServiceImpl.rollbackPoints()
    ├── Redis INCR 回滚积分
    └── 发送 [topic-author-points-consume : rollback]
```

**为什么 Redis 先扣，不直接写数据库？**
- 积分扣减是高频操作（每次调用 AI 都会触发），直接写 DB 有锁竞争和延迟
- Redis 的 `DECR` 是原子操作，天然防超扣（扣到负数时可用 Lua 脚本拒绝）
- MQ 异步落库实现最终一致性，DB 仅作持久化存储，不是实时数据源

**幂等性如何保证？**

```java
// 发送端：Redis SETNX 防重复发送
Boolean isSet = stringRedisTemplate.opsForValue()
    .setIfAbsent(idempotentKey, "1", Duration.ofHours(24));

// 消费端：DB 唯一键约束 + 捕获 DuplicateKeyException
try {
    pointsLogMapper.insert(log_FREE);  // 唯一键：idempotentKey_FREE
    pointsLogMapper.insert(log_PAID);  // 唯一键：idempotentKey_PAID
} catch (DuplicateKeyException e) {
    // 已消费过，直接跳过，幂等成功
}
```

双重幂等：**发送端防重发 + 消费端防重消费**，任何一层保护都能独立生效。

### 场景3：书架订阅通知（书籍 → 用户的跨服务解耦）

```
章节审核通过
    │
    ▼
ChapterSubmitListener 发送 [topic-book-change : chapter_update]
    │
    ▼  [user服务消费，与book服务完全解耦]
BookChangeMqListener (user服务)
    ├── 查询书架中订阅该书的所有用户
    └── 批量写入站内消息（消息内容表 + 消息接收表）
```

**为什么不用 Feign 同步调用用户服务？**
- 新章节可能有数万读者订阅，批量写入数万条消息记录可能耗时数秒
- 用 MQ 异步通知，章节发布接口立即返回，用户延迟几秒收到通知，完全可以接受
- 书籍服务不需要知道用户服务的存在，**符合开闭原则**——将来增加邮件通知、App Push，只需新增消费者，书籍服务零改动

### 场景4：ES 搜索索引同步（MQ 驱动的数据管道）

```java

// 1. 书籍审核通过
// 2. 章节发布（书籍信息变更）
// 3. 访问量定时同步（每小时批量，List<Long> 格式）
// 4. Redis写入异常时的降级兜底
```

```16:38:novel-search/novel-search-service/src/main/java/com/novel/search/listener/BookChangeMqListener.java
// ES同步消费者，失败主动抛异常触发 RocketMQ 重试
@Override
public void onMessage(MessageExt message) {
    // ...调用 Feign 查书籍 → 向量化 → 写 ES...
    // ES 写入失败时：
    throw new RuntimeException("ES sync failed, trigger retry");
}
```

**为什么 ES 同步的消费者选择抛异常触发重试，而书籍审核的消费者不抛？**

这体现了对**业务容忍度**的精准判断：
- **ES 同步失败**：搜索结果不准确，用户体验下降，**必须重试直到成功**，ES 的 `index` 操作本身是幂等的（相同 ID 覆盖写），重试安全
- **书籍入库失败**：通常是业务校验失败（书名重复等），重试没有意义，且书籍写入有唯一键，重试会一直报错进入死信队列，**不如直接 log 并人工介入**

## 五、消息可靠性的三道防线

### 第一道：发送可靠性

| 消息类型 | 发送方式 | 适用场景 |
|---|---|---|
| 普通消息 | `convertAndSend`（异步） | 绝大多数场景 |
| 批量消息 | `syncSend`（同步） | `BookVisitEsSyncJob` 每小时批量同步，需确认发送成功 |

项目未使用**事务消息**，而是用 `afterCommit()` 回调替代，原理等价但实现更轻量。

### 第二道：消费可靠性（差异化重试策略）

```
消费者是否需要重试？

积分落库/回滚 → 必须重试（资金相关，不能丢）→ throw e → RocketMQ 自动重试16次
ES 同步       → 必须重试（最终一致性要求）→ throw e → RocketMQ 自动重试16次
书籍/章节入库  → 不重试（业务校验失败）→ catch + log → 人工介入
审核结果处理   → 不重试（定时任务补偿）→ catch + log
```

### 第三道：幂等性保护

```
生产端幂等：Redis SETNX（积分场景）
消费端幂等：DB 唯一键 + DuplicateKeyException（积分日志）
          ES index 操作天然幂等（相同 bookId 覆盖写）
          业务唯一校验（书名不重复等）
```

## 六、设计不足与改进方向（面试加分项）

这些是面试官最喜欢问的"你觉得还有哪些可以改进的地方"：

**1. 无死信队列消费者**
RocketMQ 重试 16 次后消息进入 `%DLQ%topic-xxx`，但代码中没有消费死信队列。积分场景下，如果真的超过 16 次失败，这笔积分变更就永久丢失了。**改进方案**：增加死信队列消费者，触发告警并人工干预。

**2. 书籍/章节入库无幂等保护**
`BookSubmitListener` 仅靠业务校验（书名唯一），如果消息重复消费，同一本书被插入两次（如书名校验逻辑有 bug），难以发现。**改进方案**：在消息体中加入 `msgId`，入库前检查 `msgId` 是否已处理。

**3. 缺少消息轨迹和监控**
生产环境中难以追踪某条消息的完整流转链路。**改进方案**：开启 RocketMQ 消息轨迹功能（`enable-msg-trace: true`），结合 SkyWalking 做全链路追踪。

**4. 无顺序消息保证**
积分扣减和回滚如果乱序消费（回滚先于扣减处理），DB 数据会短暂出现异常。**改进方案**：对同一作者的积分消息使用顺序消息（以 `authorId` 为 MessageQueue 选择 key）。

## 面试总结话术

> 项目使用 RocketMQ 的核心目标是**三解一保**：
> - **解耦**：书籍服务不感知 AI 服务、用户服务的存在，任何下游扩展（新增通知渠道、新增数据管道）零侵入
> - **解压**：章节发布时可能触发数万读者通知，MQ 削峰将毫秒级写压力分摊到秒级消费
> - **解时**：书籍提交接口毫秒级返回，AI 审核在后台异步完成，两件事不再绑定在同一个请求响应周期里
> - **保证最终一致**：积分 Redis 先扣 + MQ 异步落库、缓存删除 + MQ 异步 ES 同步，通过 MQ 的持久化和重试机制保证数据最终一致
