# MySQL 数据库设计总结

> 本文档覆盖 turtle-website 的完整数据库设计，包括：表结构设计原则、索引优化策略、
> 反范式设计取舍、高并发写优化、分库分表落地方案，以及数据一致性保障机制。
> 每个设计决策均结合实际业务场景说明「是什么、为什么、怎么做」。

## 二、核心表结构设计

### 2.1 `book_chapter` — 核心大表，分表主体

```sql
CREATE TABLE book_chapter (
    id           BIGINT UNSIGNED,                        -- 雪花算法生成的全局唯一ID
    book_id      BIGINT UNSIGNED NOT NULL,               -- 所属书籍ID（分片键）
    chapter_num  SMALLINT UNSIGNED NOT NULL,             -- 章节号（从1开始）
    chapter_name VARCHAR(100) NOT NULL,
    word_count   INT UNSIGNED DEFAULT 0,
    content      MEDIUMTEXT NOT NULL,                    -- 章节正文（最大16MB）
    is_vip       TINYINT UNSIGNED DEFAULT 0,
    audit_status TINYINT UNSIGNED DEFAULT 0,             -- 0-待审核 1-通过 2-不通过
    audit_reason VARCHAR(500),
    create_time  DATETIME,
    update_time  DATETIME,
    PRIMARY KEY (book_id, chapter_num),                  -- 联合主键（聚簇设计）
    UNIQUE KEY uk_id (id)
);
```

**设计亮点一：联合主键 `(book_id, chapter_num)` 的聚簇效果**

InnoDB 是聚簇索引存储引擎，主键决定了数据在磁盘上的**物理排列顺序**。以 `(book_id, chapter_num)` 作为联合主键，同一本书的所有章节在 B+ 树的叶子节点上是**物理连续**的，读取某本书的全部章节目录只需一次顺序 IO 扫描，而非大量随机 IO。这等价于在分表之前对 `book_id` 做了一次天然的「软分区」，极大地提升了最高频查询的性能。

```
B+ 树叶子节点物理存储示意（按联合主键排列）：
[book_id=100, chapter_num=1] → [book_id=100, chapter_num=2] → ... → [book_id=100, chapter_num=1000]
[book_id=101, chapter_num=1] → [book_id=101, chapter_num=2] → ...
```

查询某书目录时：
```sql
SELECT chapter_num, chapter_name FROM book_chapter
WHERE book_id = 100 ORDER BY chapter_num;
-- EXPLAIN：type=ref，key=PRIMARY，Extra=Using index（索引覆盖）
```

**设计亮点二：`content` 字段存 `MEDIUMTEXT`**

章节正文与元数据（`chapter_name`、`word_count`、`audit_status`）存在同一张表，方便单次查询获取完整章节信息。`MEDIUMTEXT` 最大支持 16MB，足够容纳超长章节。但这也带来一个代价：查询章节目录（只需要 `chapter_num`、`chapter_name`）时，如果执行 `SELECT *` 会传输大量正文数据。**解决方案是在 MyBatis 中对列表查询显式指定列名**，避免 `SELECT *`：

```xml
<!-- 目录查询：只查元数据字段，不查 content -->
<select id="getChapterList" resultType="ChapterInfoRespDto">
    SELECT id, chapter_num, chapter_name, word_count, is_vip
    FROM book_chapter WHERE book_id = #{bookId} ORDER BY chapter_num
</select>
```

### 2.2 `book_info` — 书籍元数据，反范式设计的典型

```sql
CREATE TABLE book_info (
    id                       BIGINT UNSIGNED AUTO_INCREMENT PRIMARY KEY,
    work_direction           TINYINT UNSIGNED,
    category_id              BIGINT UNSIGNED,
    category_name            VARCHAR(50),         -- ★ 冗余字段（book_category.name）
    book_name                VARCHAR(50) NOT NULL,
    author_id                BIGINT UNSIGNED NOT NULL,
    author_name              VARCHAR(50) NOT NULL, -- ★ 冗余字段（author_info.pen_name）
    book_desc                VARCHAR(2000) NOT NULL,
    score                    TINYINT UNSIGNED NOT NULL,
    book_status              TINYINT UNSIGNED DEFAULT 0,
    visit_count              BIGINT UNSIGNED DEFAULT 103,
    word_count               INT UNSIGNED DEFAULT 0,
    comment_count            INT UNSIGNED DEFAULT 0,
    last_chapter_name        VARCHAR(50),          -- ★ 冗余字段（book_chapter 最新章节名）
    last_chapter_num         SMALLINT UNSIGNED,    -- ★ 冗余字段
    last_chapter_update_time DATETIME,
    is_vip                   TINYINT UNSIGNED DEFAULT 0,
    audit_status             TINYINT UNSIGNED DEFAULT 0,
    UNIQUE KEY uk_bookName_authorName (book_name, author_name),
    INDEX idx_createTime (create_time),
    INDEX idx_lastChapterUpdateTime (last_chapter_update_time)
);
```

**反范式设计原则：用冗余换查询性能**

标准第三范式（3NF）要求消除传递依赖——`author_name` 依赖于 `author_id`，应该从 `author_info` 表 JOIN 获取。但在这个小说网站的场景中，书籍列表页是访问量最大的页面，每次渲染都需要显示作者名、分类名、最新章节等信息。如果严格遵循 3NF：

```sql
-- 遵循 3NF 的查询（性能差）
SELECT b.id, b.book_name, a.pen_name, c.name, ch.chapter_name
FROM book_info b
LEFT JOIN author_info a ON b.author_id = a.id
LEFT JOIN book_category c ON b.category_id = c.id
LEFT JOIN book_chapter ch ON ch.book_id = b.id AND ch.chapter_num = b.last_chapter_num
WHERE ... ORDER BY b.visit_count DESC LIMIT 20;
```

三表 JOIN 在高并发下会对 DB 造成极大压力，且分表后跨分片 JOIN 更是噩梦。通过在 `book_info` 中冗余 `author_name`、`category_name`、`last_chapter_name`、`last_chapter_num`，书籍列表查询变成**单表操作**：

```sql
-- 反范式后的查询（性能好）
SELECT id, book_name, author_name, category_name, last_chapter_name, visit_count
FROM book_info WHERE ... ORDER BY visit_count DESC LIMIT 20;
```

**反范式的代价与应对：** 作者修改笔名时需要同步更新 `book_info.author_name`，通过异步消息队列触发批量更新，用**最终一致性**换取读性能。这是读多写少场景下的经典工程权衡。

**索引设计与查询场景的精确对应：**

- `idx_createTime`：服务「新书榜」排序查询 `ORDER BY create_time DESC LIMIT 10`，B+ 树有序，直接从最右叶子节点取10个，无需 filesort
- `idx_lastChapterUpdateTime`：服务「更新榜」排序查询，同理
- `uk_bookName_authorName`：防止同一作者重复发同名书。数据库层强制唯一性，避免应用层「先查后插」的竞态条件（先 SELECT 后 INSERT 在并发下仍可能重复）

### 2.3 `message_receive` — 覆盖索引的精妙应用

```sql
CREATE TABLE message_receive (
    id            BIGINT UNSIGNED AUTO_INCREMENT PRIMARY KEY,
    message_id    BIGINT UNSIGNED NOT NULL,
    receiver_id   BIGINT UNSIGNED NOT NULL,
    receiver_type TINYINT DEFAULT 1,               -- 0-普通用户 1-作者
    is_read       TINYINT DEFAULT 0,               -- 0-未读 1-已读
    read_time     DATETIME,
    is_deleted    TINYINT DEFAULT 0,               -- 逻辑删除
    INDEX idx_message_id (message_id),
    INDEX idx_receiver_type_id (receiver_type, receiver_id, is_read, is_deleted)  -- ★ 覆盖索引
);
```

`idx_receiver_type_id` 是一个精心设计的**覆盖索引（Covering Index）**。业务上高频查询是「统计某用户的未读消息数」：

```sql
SELECT COUNT(*) FROM message_receive
WHERE receiver_type = 1 AND receiver_id = 123 AND is_read = 0 AND is_deleted = 0;
```

这四个字段全部包含在索引中，MySQL 只需扫描二级索引的 B+ 树，**完全不需要回表**（不需要通过主键去聚簇索引读取行数据）。执行计划中 `Extra = Using index`，这是效率最高的查询方式。

**索引列顺序的设计逻辑：**

1. `receiver_type` 放首位：区分度低（只有 0 和 1），但业务查询**一定携带**此字段，作等值过滤后显著缩小范围
2. `receiver_id` 放第二位：区分度高，等值过滤后定位到具体用户的消息集合
3. `is_read`、`is_deleted` 放末位：作为过滤条件，同时使得 `COUNT(*)` 不需要回表

**逻辑删除与索引的联动：** `is_deleted` 加入索引是有原因的。逻辑删除后，被删除的记录仍然存在于数据库中，如果 `is_deleted` 不在索引中，查询时会先命中索引定位到用户的所有消息，再一条条回表判断 `is_deleted`，造成大量无效的随机 IO。

### 2.4 `content_audit` — 联合主键即幂等键

```sql
CREATE TABLE content_audit (
    id            BIGINT UNSIGNED NOT NULL,
    source_type   TINYINT UNSIGNED NOT NULL,    -- 0-书籍 1-章节
    source_id     BIGINT UNSIGNED NOT NULL,     -- 对应 book_info.id 或 book_chapter.id
    content_text  MEDIUMTEXT NOT NULL,
    ai_confidence DECIMAL(5,2),                 -- AI审核置信度，NULL表示未AI审核
    audit_status  TINYINT UNSIGNED DEFAULT 0,
    audit_reason  VARCHAR(500),
    create_time   DATETIME,
    update_time   DATETIME,
    PRIMARY KEY (source_type, source_id),       -- ★ 联合主键即幂等约束
    INDEX idx_auditStatus (audit_status)
);
```

`(source_type, source_id)` 作为联合主键，语义是「同一份内容只能有一条审核记录」，天然满足幂等性。当 MQ 消息重试导致同一内容被重复提交审核时，`INSERT INTO content_audit` 触发主键冲突，配合 `INSERT IGNORE` 或 `ON DUPLICATE KEY UPDATE` 实现无额外代码的幂等插入：

```sql
INSERT INTO content_audit (source_type, source_id, content_text, audit_status)
VALUES (1, 10086, '...', 0)
ON DUPLICATE KEY UPDATE update_time = NOW();  -- 重复插入只更新时间，不报错
```

### 2.5 `author_points_consume_log` — 多维度查询的索引策略

```sql
CREATE TABLE author_points_consume_log (
    id             BIGINT UNSIGNED AUTO_INCREMENT PRIMARY KEY,
    author_id      BIGINT UNSIGNED NOT NULL,
    consume_type   TINYINT UNSIGNED NOT NULL,    -- 0-AI审核 1-AI润色 2-AI封面
    consume_points INT UNSIGNED NOT NULL,
    points_type    TINYINT UNSIGNED NOT NULL,    -- 0-免费积分 1-付费积分
    related_id     BIGINT UNSIGNED,
    related_desc   VARCHAR(255),
    consume_date   DATE NOT NULL,
    idempotent_key VARCHAR(64),
    UNIQUE KEY uk_idempotentKey (idempotent_key),        -- ★ MQ幂等兜底
    INDEX idx_authorId (author_id),
    INDEX idx_authorId_consumeDate (author_id, consume_date),  -- ★ 联合索引
    INDEX idx_consumeDate (consume_date),
    INDEX idx_consumeType (consume_type)
);
```

**`uk_idempotentKey` 的三层幂等防护体系：**

这张表的 `idempotent_key` 唯一索引是整个 MQ 消费链路中最后一道防线：

```
第一层：Redis SETNX（author:points:deduct:idempotent:{key}，TTL=24h）
   ↓ Redis 宕机后 Key 丢失，降级到第二层
第二层：业务逻辑判断（查询数据库是否存在相同 idempotent_key）
   ↓ 极端情况下并发漏网，降级到第三层
第三层：DB 唯一索引（INSERT 触发 DuplicateKeyException，业务代码捕获后直接返回）
```

**`idx_authorId_consumeDate` 联合索引的必要性：**

最频繁的查询是「判断某作者今天是否已经免费使用了 AI 审核」（用于判断是否扣免费积分还是付费积分）：

```sql
SELECT SUM(consume_points) FROM author_points_consume_log
WHERE author_id = 123 AND consume_date = '2025-01-01' AND consume_type = 0 AND points_type = 0;
```

`idx_authorId_consumeDate(author_id, consume_date)` 命中后，精确定位到「该作者某天」的少量记录，再过滤 `consume_type`。如果只有 `idx_authorId`，需要扫描该作者的全部历史记录（可能数千条）。

## 三、高并发写优化：Write-Behind 访问量统计

### 3.1 问题背景

每次阅读章节都要更新书籍访问量：

```sql
UPDATE book_info SET visit_count = visit_count + 1 WHERE id = ?;
```

对热门书籍，每秒可能有数百次并发阅读。直接写库会导致：
- **行锁热点**：同一行被大量事务争抢，锁等待队列堆积，TPS 断崖下降
- **redo log 写放大**：每次 `UPDATE` 都要写 undo log + redo log，磁盘 IO 吃紧
- **连接池耗尽**：连接在等锁期间被占用，新请求拿不到连接，服务超时

### 3.2 解决方案：Redis 缓冲 + 定时批量回写

**写入路径**（用户访问章节时）：

```lua
-- Lua 脚本原子执行，保证三个操作要么全成功要么全失败
local zsetKey      = KEYS[1]   -- Cache::Novel::visit_rank（排行榜）
local bufferKey    = KEYS[2]   -- Cache::Novel::book_visit_buffer（写缓冲）
local bookInfoKey  = KEYS[3]   -- Cache::Novel::book_info:{bookId}（详情缓存）
local bookId       = ARGV[1]
local increment    = ARGV[2]

-- 更新访问排行榜 ZSet 分数
redis.call('ZINCRBY', zsetKey, increment, bookId)
-- 写入缓冲 Hash，等待定时任务批量回写
redis.call('HINCRBY', bufferKey, bookId, increment)
-- 如果书籍详情缓存存在，同步更新（避免缓存与排行榜不一致）
if redis.call('EXISTS', bookInfoKey) == 1 then
    redis.call('HINCRBY', bookInfoKey, 'visitCount', increment)
end
```

Lua 脚本由 Redis 单线程原子执行，三个操作不存在竞态，避免了「排行榜分数已更新但缓冲未更新」的中间态。

**回写路径**（XXL-Job 每分钟执行一次）：

```
XXL-Job 触发定时任务
    ↓
HGETALL book_visit_buffer         → 获取所有书籍的访问量增量 Map
HDEL book_visit_buffer field...   → 清除已获取的字段（先拿后删，避免丢失期间新增量）
    ↓
batchUpdateVisitCount(incrMap)    → 一条 SQL 批量更新
```

**CASE WHEN 批量更新**，将 N 次 `UPDATE` 合并为 1 次，大幅减少 DB 往返和行锁持有时间：

```sql
UPDATE book_info
SET visit_count = visit_count + CASE id
    WHEN 1001 THEN 523
    WHEN 1002 THEN 211
    WHEN 1003 THEN 88
END
WHERE id IN (1001, 1002, 1003);
```

**降级机制：** 当 Redis 不可用时，访问量更新降级到 RocketMQ 消息异步落库，保证数据不丢失：

```
Redis 写入失败
    ↓
发送 RocketMQ 消息（visitCountMq）
    ↓
消费者异步写库（单条 UPDATE，但频率低，DB 压力可接受）
```

### 3.3 数据一致性保证

Write-Behind 模式允许 DB 中的 `visit_count` 与实际值有**几分钟的延迟**，这在访问量统计场景下是可以接受的（用户看到的访问量有轻微延迟不影响体验）。读取时优先读 Redis 缓存（实时值），降级时读 DB（有延迟的值），通过缓存层屏蔽了这个不一致窗口。

## 四、分库分表设计

### 4.1 为什么 `book_chapter` 必须分表

`book_chapter` 是全站**数据量增长最快**的表：

| 指标 | 估算 |
|---|---|
| 书籍数量 | 10 万本 |
| 平均章节数 | 1000 章/本 |
| 总行数 | **1 亿行** |
| `content` 字段平均大小 | 约 6KB（3000字 × 2字节） |
| 总存储量 | **约 600GB** |

单表 1 亿行时，InnoDB B+ 树深度达 5-6 层，每次叶子节点查询需要 5-6 次磁盘 IO。`content` 字段的 `MEDIUMTEXT` 导致行数据极大，页分裂频繁，写入性能严重劣化。

### 4.2 分片键选择：`book_id`

**原设计的错误（`chapter_id % 10`）：**

按 `chapter_id` 分表，同一本书的章节被随机散落到 10 张分表中。查询「某本书的目录」时，ShardingSphere 无法确定数据在哪张分表，只能**向所有分表广播查询（Scatter-Gather）**，再在内存中合并结果：

```
查询：WHERE book_id = 100
  → 广播到 book_chapter_0：SELECT ... WHERE book_id = 100  （可能无结果）
  → 广播到 book_chapter_1：SELECT ... WHERE book_id = 100  （可能有部分章节）
  → ...
  → 广播到 book_chapter_9：SELECT ... WHERE book_id = 100  （可能有部分章节）
  → 汇总合并结果
```

10 次并行查询 + 内存归并，比不分表还慢。

**正确的分片键：`book_id`**

以 `book_id` 为分片键，同一本书的所有章节路由到同一张分表，每次查询**精确命中单一分片**：

```
查询：WHERE book_id = 100
  → 计算路由：HASH(100) % 8 = 4
  → 仅查询 book_chapter_4
  → 直接返回，无 Scatter-Gather
```

选择 `book_id` 作为分片键有三重理由：

1. **查询路由精确**：所有业务查询（目录、正文、字数统计、审核状态更新）均携带 `book_id`，100% 的查询能精确路由
2. **数据局部性**：同一本书的章节在分表内仍然按聚簇索引 `(book_id, chapter_num)` 物理有序，顺序 IO 优势得以保留
3. **写入局部性**：作者写章节时，同一本书的写入只打一张分表，行锁竞争范围收窄

### 4.3 分表数量：8 张

选择 8 而非 10 的原因：

- **2 的幂次**：未来数据量翻倍需要扩容时，从 8 张扩到 16 张，只需将每张分表的**一半数据**迁移到新表（按 `HASH(book_id) % 16 >= 8` 过滤），无需全量重 Hash
- **容量预估**：1 亿行 ÷ 8 = 1250 万行/表，B+ 树深度约 4 层，单次查询 4 次磁盘 IO，性能达标
- **连接数可控**：Scatter-Gather 时最多同时打开 8 个分片连接，配合 `max-connections-size-per-query: 1`，不会耗尽连接池

### 4.4 分片算法：HASH_MOD

```yaml
shardingAlgorithms:
  bookChapterSharding:
    type: HASH_MOD      # 哈希取模，而非 INLINE 纯取模
    props:
      sharding-count: '8'
```

**为什么不用 `INLINE`（`book_id % 8`）？**

纯取模在 `book_id` 分布不均匀时会造成**数据倾斜**。本项目的 `book_id` 由数据库自增生成，早期平台书籍 ID 集中在小值（如 1～10000），`id % 8` 的结果中 0、1、2、3 号分表会有更多书籍的早期历史数据，分布不均。`HASH_MOD` 先对 `book_id` 做哈希散列再取模，将 `book_id` 的分布从局部集中变为全域均匀。

### 4.5 全局唯一 ID：雪花算法（Snowflake）

分表后，`AUTO_INCREMENT` 只能保证单表内自增，跨分表会出现重复 ID（如 `book_chapter_0` 和 `book_chapter_1` 各自都有 `id=1`）。必须引入全局唯一 ID 生成方案。

```yaml
keyGenerators:
  snowflake:
    type: SNOWFLAKE
    props:
      worker-id: 1    # 多实例部署时需保证唯一（建议通过 Nacos 或 Redis 动态分配）
```

**雪花 ID 的 64 位结构：**

```
┌──────────────────────────────────────────────────────────────────────┐
│ 0 │          41位时间戳（毫秒）         │  10位机器ID  │  12位序列号  │
└──────────────────────────────────────────────────────────────────────┘
  1位   ← 符号位，永远为0                                    ↑
  符号                                               每毫秒最多4096个
```

雪花 ID 的三大优势：
1. **趋势递增**：时间戳在高位，ID 随时间单调增大，对 B+ 树的**顺序插入友好**，避免随机写导致的页分裂（页分裂是 InnoDB 写性能的主要杀手之一）
2. **高性能**：纯内存计算，每毫秒每节点最多生成 4096 个 ID，无需访问数据库或 Redis
3. **可排序**：按 ID 排序近似等于按创建时间排序，业务上可以直接用 ID 做游标分页

**Worker ID 的分配策略：** 多实例部署时（如 `novel-book-service` 部署了 3 个节点），每个节点的 `worker-id` 必须不同，否则同一毫秒内的 ID 会重复。推荐在服务启动时向 Redis 执行 `INCR snowflake:worker-id`，将获取到的值（取模10位范围）作为本节点的 `worker-id`，并设置适当 TTL 实现自动回收。

### 4.6 广播表（Broadcast Table）

```yaml
broadcastTables:
  - book_info
  - book_category
  - home_book
```

`book_chapter` 分表后，业务查询中存在与 `book_info` 的 JOIN：

```sql
-- 查章节详情时需要书名（用于消息推送文案）
SELECT ch.chapter_name, b.book_name
FROM book_chapter ch JOIN book_info b ON ch.book_id = b.id
WHERE ch.book_id = 100 AND ch.chapter_num = 1;
```

如果 `book_info` 是普通单表，这条 SQL 需要跨节点——先从 `book_chapter_4`（book_id=100 路由到分片 4）读取章节，再去另一个节点读取 `book_info`。

**广播表的原理：** ShardingSphere 在每个分片节点上都维护 `book_info` 的一份全量副本，写入时同步广播到所有节点，读取时直接从本地副本 JOIN，彻底消除跨节点查询：

```
写入 book_info（如新书通过审核）：
  → ShardingSphere 广播：同步写入 ds_0 上所有物理节点的 book_info 副本

查询（JOIN book_chapter_4）：
  → 直接与 ds_0 本地的 book_info 副本 JOIN，无需跨节点
```

广播表适用于：**数据量小、读多写少、被频繁 JOIN** 的配置型/字典型表。`book_category`（书籍分类）和 `home_book`（首页推荐）同理。

### 4.7 完整 ShardingSphere 配置

```yaml
# sharding-config.yaml（ShardingSphereDriver 模式）
dataSources:
  ds_0:
    dataSourceClassName: com.zaxxer.hikari.HikariDataSource
    jdbcUrl: jdbc:mysql://.../turtle_website?...
    maximumPoolSize: 15

  - !SHARDING
    tables:
      book_chapter:
        actualDataNodes: ds_0.book_chapter_$->{0..7}   # 8张分表
        tableStrategy:
          standard:
            shardingColumn: book_id                      # 分片键
            shardingAlgorithmName: bookChapterSharding
        keyGenerateStrategy:
          column: id
          keyGeneratorName: snowflake                    # 全局唯一ID

    broadcastTables:                                     # 广播表
      - book_info
      - book_category
      - home_book

    shardingAlgorithms:
      bookChapterSharding:

        type: SNOWFLAKE

```

```yaml

    url: jdbc:shardingsphere:classpath:sharding-config.yaml
```

### 4.8 分表后的注意事项

**① 跨分片查询的处理原则**

管理后台查询「所有待审核章节」时，没有携带 `book_id`，会触发 Scatter-Gather（向 8 张分表广播查询）。对策：
- 管理后台查询频率极低（审核是低频操作），接受 Scatter-Gather
- 或通过 `content_audit` 表（单表，存有 `source_id=chapter_id`）先过滤，再反查对应 `book_id` 做精确路由

**② 跨分片分页的陷阱**

全局分页（不带 `book_id`）在分表环境下存在深翻页问题：

```sql
-- 危险：每张分表都要取 10010 条，内存排序后取 10 条
SELECT * FROM book_chapter ORDER BY create_time LIMIT 10000, 10;
```

解决方案：**游标分页（Keyset Pagination）**，利用雪花 ID 的趋势递增特性：

```sql
-- 安全：每张分表只需取 id > lastId 的前 10 条，归并后总量可控
SELECT * FROM book_chapter WHERE id > #{lastId} ORDER BY id LIMIT 10;
```

本项目 XXL-Job 全量同步 ES 的定时任务已经采用游标分页，这正是这种模式的典型应用。

**③ 数据迁移（存量数据迁移到分表）**

线上系统切换分表需要零停机迁移，推荐**双写迁移方案**：

```
阶段1（双写）：
  新请求同时写旧单表 + 新分表
  读仍从旧单表读（保证回滚能力）
  持续 24～48 小时，等存量数据迁移完成

阶段2（存量迁移）：
  后台任务按 book_id 范围批量读旧表，写入对应分表
  用 Canal 订阅 binlog 做增量追赶，确保新旧表一致

阶段3（切读）：
  读流量切换到分表（1% 灰度 → 10% → 100%）
  观察错误率和延迟，异常时快速回切

阶段4（下线旧表）：
  停止双写，旧表归档
```

## 五、事务与数据一致性

### 5.1 事务内 MQ 消息的经典陷阱

章节发布时，需要写库并发送 MQ 消息（触发 AI 审核、更新 ES 等）。错误写法：

```java
@Transactional
public void publishChapter(Chapter chapter) {
    chapterMapper.insert(chapter);        // DB 写入（事务中）
    mqProducer.send(chapterSubmitMq);     // ← 危险！此时事务尚未提交
    // 若后续操作抛异常导致事务回滚，DB 无数据，但 MQ 消息已发出
    // 消费者（AI审核服务）会处理一个不存在的章节
}
```

**本项目的正确做法：** 通过 `TransactionSynchronizationManager` 注册事务提交后的回调：

```java
@Transactional
public void publishChapter(Chapter chapter) {
    chapterMapper.insert(chapter);
    // 事务提交成功后才触发 MQ 发送，保证 DB 有数据 MQ 才有消息
    TransactionSynchronizationManager.registerSynchronization(
        new TransactionSynchronization() {
            @Override
            public void afterCommit() {
                mqProducer.send(chapterSubmitMq);
            }
        }
    );
}
```

`afterCommit()` 在事务提交完成后才执行，即使 MQ 发送失败，DB 数据已经持久化，可以通过重试恢复；而 DB 事务回滚时 MQ 消息不会发出，避免了「消费者处理幽灵数据」的问题。

### 5.2 三层幂等保障

MQ 消费端（如积分扣减）通过三层递进的幂等机制防止重复处理：

```
消费者接收到消息
    ↓
第一层：Redis SETNX
  SET author:points:deduct:idempotent:{key} 1 EX 86400 NX
  → 返回 1：首次处理，继续
  → 返回 0：已处理过，直接 ACK 返回（Redis 宕机时 Key 丢失，降级到第二层）
    ↓
第二层：DB 查询（double check）
  SELECT id FROM author_points_consume_log WHERE idempotent_key = #{key}
  → 无结果：继续
  → 有结果：已处理过，直接 ACK 返回
    ↓
第三层：DB 唯一索引兜底
  INSERT INTO author_points_consume_log (..., idempotent_key)
  → 成功：正常处理
  → 抛出 DuplicateKeyException：已处理过，捕获异常后直接 ACK 返回
```

三层防护保证了即使在 Redis 宕机、并发竞争等极端情况下，积分扣减也不会重复执行。

## 六、连接池调优

```yaml
hikari:
  minimum-idle: 5
  maximum-pool-size: 15      # 每个微服务节点最多 15 个连接
  connection-timeout: 30000  # 获取连接等待上限 30s，超时抛异常而非无限等待
  idle-timeout: 600000       # 空闲连接超过 10min 自动释放（降低 MySQL 端无用连接占用）
  max-lifetime: 1800000      # 连接最大存活 30min（早于 MySQL wait_timeout，主动重建）
  leak-detection-threshold: 60000  # 连接被持有超过 60s 则打印警告日志（排查连接泄漏）
  connection-test-query: SELECT 1
  pool-name: NovelHikariPool
```

**`maximum-pool-size: 15` 的依据：**

HikariCP 官方给出的公式：`pool_size = (core_count × 2) + effective_spindle_count`

对于典型的 4 核云服务器，理论值约为 9。设置为 15 是保留了一定余量，同时考虑到微服务架构下的全局连接数：4 个微服务 × 15 = 60 个并发连接，需要确保 MySQL 的 `max_connections` 设置在 100 以上并留有余量（监控、运维工具也需要连接）。

**`max-lifetime: 1800000`（30分钟）的意义：**

MySQL 默认 `wait_timeout = 8小时`，长时间空闲的连接会被中间层（负载均衡、NAT 网关）的防火墙规则强制关闭，但 MySQL 和连接池都不知道这个 TCP 连接已经断开。下次使用这个「僵尸连接」时，会收到 `CommunicationsException: The last packet sent successfully to the server was 0 milliseconds ago`。将 `max-lifetime` 设置为 30 分钟，连接池主动在防火墙超时前回收重建，彻底避免僵尸连接问题。

## 七、设计决策速查表

| 设计点 | 方案 | 核心原因 |
|---|---|---|
| `book_chapter` 主键 | `(book_id, chapter_num)` 联合主键 | 聚簇存储使同书章节物理连续，顺序 IO |
| `book_info` 冗余字段 | 反范式，冗余 `author_name` 等 | 书籍列表是读密集场景，消除 JOIN |
| 分表对象 | `book_chapter` | 全站最大表，内含 `MEDIUMTEXT` 大字段 |
| 分片键 | `book_id`（而非 `chapter_id`）| 所有查询必带 `book_id`，精确路由 |
| 分片算法 | `HASH_MOD` | 分布均匀，避免 `book_id` 小值集中导致倾斜 |
| 分表数 | 8（2 的幂次） | 扩容时只需迁移一半数据，无需全量 Rehash |
| 全局 ID | 雪花算法 | 趋势递增（B+ 树友好），高性能，无需 DB |
| 广播表 | `book_info`/`book_category`/`home_book` | 消除跨分片 JOIN，本地副本直接关联 |
| 访问量写入 | Write-Behind（Redis 缓冲 + 定时批量写回）| 消除高频写热点，避免行锁竞争 |
| 消息幂等 | Redis SETNX + 业务判断 + DB 唯一索引 | 三层递进防护，极端情况兜底 |
| 事务与 MQ | `afterCommit()` 钩子发消息 | 保证事务提交后 MQ 才发出，避免幽灵数据 |
| 逻辑删除 | `is_deleted` 字段 + 纳入复合索引 | 可追溯，避免 B+ 树碎片，索引消除回表 |
| 连接池 | HikariCP，`max-lifetime=30min` | 防僵尸连接，连接数与 MySQL 容量匹配 |
