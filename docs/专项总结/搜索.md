

# 项目搜索功能设计深度总结（面试专用）

---

## 一、为什么用 Elasticsearch？（不用 MySQL like 查询）

**核心矛盾**：小说平台的搜索场景是典型的**全文检索**需求，MySQL 的 `LIKE '%关键词%'` 有三大致命缺陷：

| 对比维度 | MySQL LIKE | Elasticsearch |
|---------|-----------|--------------|
| 中文分词 | 不支持，无法理解"修仙"≠"修""仙" | IK Analyzer 专业中文分词 |
| 性能 | 全表扫描，索引失效，百万级数据秒级响应不了 | 倒排索引，毫秒级响应 |
| 相关性排序 | 只能 0/1 匹配，无相关性概念 | BM25 算法计算相关性分数 |
| 语义理解 | 完全不支持 | 支持向量检索语义匹配 |
| 高亮显示 | 需要应用层处理 | 原生支持高亮 |

**深层原因**：Elasticsearch 底层是 Lucene，核心是**倒排索引**结构——将文档内容分词后建立"词→文档列表"的映射，使得关键词查找的时间复杂度从 O(N) 降为近似 O(1)，这是它能在海量数据上做全文检索的根本原因。

## 二、ES 索引设计——每个决策都有理由

索引 Mapping 的核心字段设计体现了多个工程权衡：

### 2.1 `bookName` 双类型字段设计

```json
"bookName": { 
  "type": "text", 
  "analyzer": "ik_smart",
  "fields": { 
    "keyword": { "type": "keyword" } 
  } 
}
```

**为什么这么设计？**
- `text` 类型：用于**分词全文检索**，"修仙小说"会被分成["修仙","小说"]，支持模糊搜索
- `keyword` 子字段：用于**精确匹配和通配符**，保存原始未分词的字符串
- **实际作用**：搜索模式 2（书名精准搜）时，代码同时用了 `match`（分词）和 `wildcard`（通配符）两路，解决了中英文混合书名（如"God of War"）分词失效的问题

### 2.2 IK 分词器的索引时/搜索时差异

- **索引时**：`bookDesc` 用 `ik_max_word`（细粒度，最大切分），"修仙小说"→["修","仙","修仙","小说"]，**提高召回率**
- **搜索时**：`ik_smart`（智能粗粒度），"修仙"→["修仙"]，**提高精确率**
- **本质**：这是一个召回率与精确率的权衡——宁可多召回，再通过 BM25 分数筛选

### 2.3 向量字段设计

```json
"embedding": {
  "type": "dense_vector",
  "dims": 1024,
  "index": true,
  "similarity": "cosine"
}
```

- `dims: 1024`：与 DashScope text-embedding-v4 输出维度严格一致，维度不匹配会报错
- `index: true`：告诉 ES 为向量建立 **HNSW 索引**，否则 KNN 查询会退化为全量暴力扫描
- `similarity: cosine`：选余弦相似度而非欧氏距离（原因见下文）

### 2.4 `picUrl` 不建索引

```json
"picUrl": { "type": "keyword", "index": false }
```

图片 URL 不参与任何搜索，设置 `index: false` 节省存储空间和索引开销——这是**知道什么不需要索引**的体现。

## 三、向量数据库设计——为什么 ES 而非专用向量数据库？

### 3.1 选型决策

这里有个关键技术决策：**为什么不用 Milvus、Pinecone 等专用向量数据库？**

理由很充分：
1. **避免系统复杂性**：已经部署了 ES，再加向量数据库需要维护两套存储、两套数据同步
2. **ES 向量能力已够用**：ES 8.x 的 `dense_vector` + HNSW 索引，KNN 查询性能足够应对中小规模
3. **混合查询天然支持**：向量过滤条件（`filter`）和文本过滤可以在一个查询里完成，跨系统 join 实现不了这个

### 3.2 向量化内容的设计

```java
String textToEmbed = "书名:" + book.getBookName() + 
                   "; 作者:" + book.getAuthorName() + 
                   "; 简介:" + book.getBookDesc();
```

**为什么这么拼接？**
- **结构化前缀**（"书名:"、"作者:"）：帮助模型理解字段语义，而不是把所有字段混在一起
- **字段顺序隐式权重**：书名在最前面，模型对文本开头的权重更大
- **截断到 2000 字符**：text-embedding-v4 支持 8192 tokens，但简介过长时后半段语义对搜索贡献有限，截断既省 API 费用，又保证核心语义

### 3.3 为什么用余弦相似度而非欧氏距离？

**这是面试高频考点：**

文本向量的语义信息体现在**方向**上，而非**长度**上。
- 例："修仙小说"和"修真小说"的向量方向高度一致，但向量模长（L2 范数）可能不同
- 余弦相似度 = 点积 / (A的模 × B的模)，**自动归一化**，只看角度
- 欧氏距离受向量长度影响，短书名和长简介的向量长度差异大，会造成误判

## 四、搜索算法设计——三档智能搜索模式

这是整个搜索系统最核心的设计，代码体现了清晰的策略模式：

### 4.1 三种模式

```java
int mode = condition.getSearchMode() == null ? 0 : condition.getSearchMode();
if (mode == 1) {
    useHybridSearch = true;         // 强制混合搜索
} else if (mode == 0) {
    // 智能自动切换：关键词 > 6 字 或 含空格 → 混合搜索
    if (condition.getKeyword().length() > 6 || condition.getKeyword().contains(" ")) {
        useHybridSearch = true;
    }
}
// mode == 2：强制文本搜索（书名精准搜）
```

**设计思想**：
- **模式 0（智能自动）**：核心启发是"短词是关键词，长句是语义查询"。"斗破苍穹"→文本搜索；"讲述一个从农村出来的孩子逆袭成神的故事"→混合搜索。这避免了对简单书名搜索做不必要的 API 调用（省成本，降延迟）
- **模式 1（强制混合）**：用户主动选择"搜情节"场景
- **模式 2（精准书名）**：解决中英文混合书名分词失效问题

### 4.2 混合搜索的结果融合——手动 RRF

```java
// KNN 召回 → 按排名计分
double knnScore = 1.0 / (i + 1.0);   // 第1名得 1.0，第2名得 0.5...

// BM25 召回 → 按排名计分  
double bm25Score = 1.0 / (i + 1.0);

// 加权合并
综合分数 = knnScore × 0.6 + bm25Score × 0.4
```

**为什么用排名倒数分，而不是直接用原始分数？**
- KNN 输出余弦相似度（0~1），BM25 输出的是 TF-IDF 分数（可以很大），量纲完全不同，不能直接相加
- 排名倒数分（RRF 思想）将两路分数统一到可比较的区间，是工业界经典解法

**为什么向量权重 0.6，文本权重 0.4？**
- 该场景是"搜情节/搜语义"，向量理解更准确，所以权重更高
- 文本权重保留是为了防止向量产生语义漂移（极端情况下向量检索可能找到语义相近但内容无关的书）

**为什么不用 ES 原生 RRF？**
- ES 原生 RRF 需要白金/企业级商业许可证，开源版不支持
- 手动实现虽然需要两次网络往返 + 内存合并，但在搜索服务不是超高 QPS 的场景下完全可接受
- 优势：权重可调，可根据 A/B 测试结果灵活调整

### 4.3 多字段权重查询

```java
b.must(q -> q.multiMatch(t -> t
    .fields(
        EsConsts.BookIndex.FIELD_BOOK_NAME + "^2",      // 书名权重×2
        EsConsts.BookIndex.FIELD_AUTHOR_NAME + "^1.8",  // 作者权重×1.8
        EsConsts.BookIndex.FIELD_BOOK_DESC + "^0.1"     // 简介权重×0.1
    )
    .query(condition.getKeyword())
));
```

**为什么简介权重这么低（0.1）？**
- 简介内容更长、词更散，BM25 的 TF 值相对更高，不压低会导致简介偶尔提到某词的书被过度排名
- 书名和作者匹配才是用户真正想要的精确信号

## 五、数据同步机制——双通道保障数据最终一致性

### 5.1 实时同步：MQ 驱动（事件驱动架构）

```
书籍更新/新增 → 发送 RocketMQ 消息(bookId) 
               → BookChangeMqListener 消费 
               → Feign 拉取最新数据 
               → 生成向量 
               → 写入 ES
```

**为什么不直接在书籍服务里写 ES？**
- **解耦**：书籍服务不应该感知搜索服务的存在，这是微服务边界的基本原则
- **可靠性**：MQ 提供消息持久化和重试，避免直接调用时因网络问题丢数据
- **异步**：写 ES 是耗时操作（还要调外部 API 生成向量），放 MQ 异步处理不阻塞主流程

**删除逻辑的精细设计**：

```java
if (book == null) {
    if (existsInEs(bookId)) {
        // ES 有数据 → 可能是删除/未审核通过 → 从 ES 删除
        deleteFromEs(bookId);
    } else {
        // ES 没有数据 → 可能是新增但 Feign 调用失败 → 跳过，不做删除
        log.warn("ES 中不存在该书籍，跳过处理");
    }
}
```

这里区分了两种 null 情况，**避免了误删**——如果新增书籍时 Feign 调用失败，不应该去 ES 删一个本来就不存在的数据（也删不掉，但避免了无意义的 ES 操作和错误日志）。

### 5.2 全量同步：游标分批 + Bulk API

```java
for (;;) {
    List<BookEsRespDto> books = bookFeignManager.listEsBooks(maxId); // 每批30条，游标翻页
    if (books.isEmpty()) break;
    
    BulkRequest.Builder br = new BulkRequest.Builder();
    for (BookEsRespDto book : books) {
        // 生成向量...
        br.operations(op -> op.index(idx -> idx.document(book)));
        maxId = book.getId();
    }
    elasticsearchClient.bulk(br.build()); // Bulk 批量写入
}
```

**为什么用游标分页而不是 offset 分页？**
- `LIMIT offset, size` 在大 offset 时性能急剧下降（需要扫描前 offset 条记录）
- 游标分页（`WHERE id > maxId LIMIT size`）始终利用主键索引，性能稳定

**为什么用 Bulk API 而不是逐条写入？**
- 每次 ES 写入都涉及网络往返 + 倒排索引更新
- Bulk API 合并多次操作为一个请求，降低网络开销，吞吐量提升 5-10 倍

## 六、高可用设计——降级保护

```java
try {
    // 生成向量 → 混合搜索
    return hybridSearch(condition);
} catch (Exception e) {
    log.error("向量生成或混合检索失败，降级为普通文本搜索", e);
    return executeTextOnlySearch(condition, boolQuery); // 降级
}
```

**降级链路**：
1. DashScope API 超时/限流 → 自动切换纯 BM25 文本搜索
2. 书籍 MQ 向量生成失败 → 书籍元数据仍写入 ES，只是 `embedding` 字段为空，不影响文本搜索

这是**可用性优先于完美性**的工程哲学体现——搜索结果质量稍差，总比搜索服务不可用要好。

## 七、面试可能追问的深度问题

**Q：如何保证 ES 和 MySQL 的数据一致性？**
> 通过 MQ 异步同步，最终一致性。如果 MQ 消费失败，RocketMQ 的重试机制（默认重试 16 次）会保证消息最终被处理。同时保留全量同步接口，作为数据修复兜底手段。

**Q：向量搜索为什么需要设置 numCandidates？**
> HNSW 是近似最近邻算法，`numCandidates` 是探索的候选集大小，越大精度越高但速度越慢。这是精度与性能的权衡，设为 `max(recallSize * 2, 200)` 保证足够的候选集同时控制性能。

**Q：如何评估搜索质量？**
> 技术指标：P50/P95/P99 响应时间、向量生成耗时、KNN/BM25 各路召回数量比例。业务指标：点击率（CTR）、平均点击位置（越小越好，< 3 为目标）、无结果率（< 5%）。

**Q：未来最值得做的优化是什么？**
> 最高性价比的是**查询向量缓存**（Redis TTL 1 小时），因为用户搜索词有很强的重复性，可以减少 50-80% 的 DashScope API 调用，延迟降低约 200ms，实现代价很低。
