# 章节审核：会话续写与摘要交接 — 落地预案

> **状态**：预案（未实现）。与当前代码对齐：`ChapterSegmentStep` 分段 → `ChapterSegmentAuditStep` 逐段单次调用 LLM；书籍/章节共用审核并发闸门见 `novel.ai.audit.max-concurrent`。

---

## 一、目标与边界

| 目标 | 说明 |
|------|------|
| **同一会话多段** | 在上下文允许时，多段正文在同一条「对话」中顺序审核，模型可引用前文。 |
| **触顶则交接** | 预估超出单次对话安全余量时，**自动**结束当前会话，生成 **审核向前文摘要**，在新会话中继续审下一段。 |
| **可观测、可回滚** | 行为由配置与版本化 Prompt 控制；出问题可一键退回「每段独立调用」模式。 |

| 非目标（首期可不做的） | 说明 |
|-------------------------|------|
| 改书籍审核单段形态 | 仅章节多段路径。 |
| 替代现有结构化输出 | 每段仍输出 `AuditDecisionAiOutput` 同类结构，便于合并与存量逻辑兼容。 |

---

## 二、现状与差距

**现状**：每段独立构造 system/user → `StructuredOutputInvoker` 一次 → 无 Chat Memory、无跨段对话状态。

**差距**：需 **会话状态**（消息列表或等价物）、**长度/Token 预算**、**摘要子流程**、**新会话首条消息拼装规则**，以及 **失败降级策略**。

---

## 三、总体方案（概念架构）

```
分段列表 segments[0..n-1]
  ↓
for 每段 i:
  1) 评估：当前会话累计上下文 + 本段是否超过阈值？
  2) 若超过 → 调用「摘要步骤」→ 清空/归档会话 → 新会话首包注入：系统说明 + 摘要 + 可选「上段结论要点」
  3) 若不超 → 在同一会话追加 user（本段 + 审核指令）
  4) 调用 LLM（结构化输出）→ 写入 segmentResults[i]
  5) 异常策略保持与现网一致（内容安全短路 / 段级待审核）
```

**「Agent 自行选择」**：首期建议 **规则为主、模型为辅** —— 硬阈值必切会话；边界情况可插入一次 **轻量规划调用**（可选，二期）。

---

## 四、与现有流水线如何衔接

| 模块 | 改动思路 |
|------|-----------|
| `ChapterAuditContext` | 增加会话侧字段（见下节）；可选记录「当前第几轮会话」「已用 token 估计值」。 |
| `ChapterSegmentAuditStep` | 从「每段全新 prompt」改为「会话驱动」：内部持有一个 **会话执行器**（新建类，避免 Step 过胖）。 |
| `StructuredOutputInvoker` | 保留；每次 user 仍走结构化输出；若 Spring AI 需 **带 history 的 ChatClient 调用**，可扩展 invoker 或增加 **带 Message List 的重载**（实现时二选一）。 |
| RAG | 明确策略：**每段仍用当前段正文做 query** 召回（与现网一致），摘要仅进会话不占 RAG query；或配置「仅首轮召 RAG」。 |
| Prompt | 新增 `.st`：`chapter-audit-session-summary`（摘要专用）、可选 `chapter-audit-handoff-system`（新会话说明）；原 `chapter-audit` 增加占位符：`rollingSummary`、`sessionSegmentRange`。 |
| 配置 | `novel.ai.chapter.audit.session.*`：开关、单会话最大字符/token、摘要模型是否同主模型、是否启用规划调用等。 |

## 五、上下文模型（建议字段）

在 `ChapterAuditContext` 中扩展（命名可再斟酌）：

- `rollingAuditSummary`：String，跨段累积的 **审核向摘要**（每次交接更新）。  
- `currentSessionId` 或内部 `List<Message>`：若用 Spring AI `ChatMemory` 则存 conversationId；若手写 messages 则存在上下文对象中（注意线程安全：单请求单上下文，无跨请求共享）。  
- `estimatedPromptChars` / `estimatedTokens`：粗算即可（字符 × 系数 或 tiktoken 类库若引入）。  
- `sessionRound`：第几次「新会话」，便于日志与指标。

**不**建议把整章前文重复拼进每一段 user（会快速触顶）；触顶后靠 **摘要 + 下一段正文**。

---

## 六、触发条件（首期可落地）

1. **主阈值**：`当前会话内累计字符数 + 本段字符 + 系统模板固定开销 > novel.ai.chapter.audit.session.max-chars`（或 token 上限）。  
2. **段数兜底**：同一会话最多包含 `max-segments-per-session` 段，防止段很短但轮次极多。  
3. **摘要长度上限**：摘要输出 `max-summary-chars`，超出则二次压缩或截断（需标风险）。

**二期可选**：模型输出 JSON `{ "action": "CONTINUE" | "HANDOFF", "reason": "..." }`，仅当粗算在边界 80%～100% 时调用，减少误切。

---

## 七、摘要子流程（审核专用）

**输入**：当前会话内 **已审段** 的要点（可从每段结构化结果抽取 `auditStatus/reason` 关键词）+ 必要时 **末段正文截断**。  

**输出**：严格 Prompt 约束：**保留合规风险点、敏感倾向、未闭合情节指代、与下段可能相关的禁令**；避免纯剧情复述占满篇幅。  

**实现**：单独一次 `chatClient` 调用（可用无工具、小温度），**不走 RAG** 或可选轻量 RAG；结果写入 `rollingAuditSummary`，然后 **清空当前 Chat Memory / 消息列表**，进入新会话。

---

## 八、合并与整章结果

- **段级结果**：仍写入 `segmentResults`，与现网一致。  
- **整章合并**：沿用现有 `ChapterResponseBuildStep` 逻辑；若合并规则需考虑「同会话多段」，仅当 **段级 JSON 结构不变** 时，合并不需要改或小幅调整。  
- **短路**：内容安全仍 **整章不通过**，逻辑保持不变。

---

## 九、风险与对策

| 风险 | 对策 |
|------|------|
| 摘要丢失关键违规信息 | Prompt 强约束 + 段级结构化结论一并传入摘要步骤；高危段可禁止摘要合并（配置黑名单关键词触发整段保留）。 |
| Token 估算不准 | 留 10%～20% 安全余量；触顶略保守。 |
| 成本上升 | 仅多段且触顶才摘要；开关默认关或仅灰度类别。 |
| 调试困难 | `sessionRound`、每段 `handoff=true/false` 打日志；Micrometer counter：`chapter.audit.session.handoff`。 |
