## 一、当前 novel-ai 模块诊断（和"标准 Agent"的差距）

你现在的实现本质上是 **"Spring AI + 阿里百炼 DashScope 的 Prompt 裸调用"**，它能跑通业务，但离 "企业级 Agent" 还缺以下核心要素（以 `TextServiceImpl` 为样本逐点对比）：

### 1) 缺少"Agent"骨架，只有 Service 直接调 Chat
- 所有审核逻辑都在一个 `TextServiceImpl` 里：`auditBook/auditChapter` 等直接拼 Prompt → 调 `chatClient` → 正则 + Jackson 兜底解析。
- 没有 **Tool/Function Calling**、**Planner-Executor**、**Multi-step**、**Memory** 这些 Agent 标配。
- 审核流程是"一次 LLM 调用决定一切"，没有**多维度独立判定 + 聚合决策**，也没有**人机协同回路**（HITL）。

### 2) Prompt 工程不规范
- Prompt 用 `String.format` 硬编码在 Java 里（`buildAuditPrompt`、`buildChapterAuditPrompt`、`buildPolishPrompt`…），这会导致：
    - 无法版本化 / A-B 测试 / 热更新
    - 审核标准写死在代码里，业务或法务要调整时必须改代码发版
    - 难以做 Few-shot、Chain-of-Thought、Reflection 等高级提示策略
- 对比 `interview-guide-master`：
    - 用 `.st`（StringTemplate）模板文件放在 `resources/prompts/`
    - `PromptTemplate` + `ResourceLoader` 加载渲染
    - 通过配置项（`ResumeAnalysisProperties`）控制路径、参数

### 3) 结构化输出不规范（最大痛点之一）
你现在的做法：
- Prompt 里"请求 JSON" → `extractJsonFromResponse` 用正则抠 JSON → `extractField` 用 Jackson+正则兜底。
- 失败后 fallback 成"待审核"，损失严重。

标准做法（`interview-guide-master` 已在用）：
- `BeanOutputConverter<T>` + `.entity(converter)`
- 把格式指令 `outputConverter.getFormat()` 追加到 system prompt
- 失败自动重试（见它的 `StructuredOutputInvoker`，带 `app.ai.structured_output.*` 埋点和 "repair prompt"）

### 4) 异常处理脆弱
- `isContentInspectionFailed(e)` 用字符串 `contains("datainspectionfailed")` 判断 DashScope 的内容安全错误 —— 一旦厂商文案变化或切换供应商就失效。
- 没有区分"可重试异常"（限流、超时）与"不可重试异常"（参数错误、内容违规），重试逻辑只在 `ImageServiceImpl` 的 `@Retryable` 上，`auditBook/auditChapter` 没有。

### 5) RAG 实现比较粗糙
- `getSimilarAuditExperiences` 通过 Feign 调 `novel-search-service`，根据 `topK=3, threshold=0.75` 拿"历史审核判例"。
- 但：
    - 没看到 **embedding 管理**（向量化模型选型、向量库、维度、重新 embed 的机制）
    - 没有 **query rewrite**（对比参考项目的 `rewritePromptTemplate`），审核文本经常很长，直接去检索召回率差
    - 没有 **chunking 策略**（参考项目有短/中/长问题的 topK / minScore 分级）
    - 没有 **Rerank**（BGE-Reranker 等二次排序）

### 6) 内容分段（chunking）不够鲁棒
`splitContent` 只按字符数 + 标点断句，没有：
- **overlap**（相邻段落重叠 100~200 字），避免语境被切断
- **语义切分**（按段落/场景/对话体切分）
- **分段并发审核**（目前是串行 for-loop，性能差、也没有熔断）

### 7) 观测 / 治理只做了一半
- SkyWalking 自定义 Span 写得不错（`ActiveSpan.tag`），但：
    - 没有 **token 消耗** 埋点（关键成本指标）
    - 没有 **模型版本 / 供应商** 统一埋点
    - 没有 **命中策略 / 召回命中率 / 审核通过率** 的业务指标
    - 没有 **Prompt hash / 内容 hash** 这类可复现性埋点
- 没有 Micrometer Metrics（参考项目 `StructuredOutputInvoker` 里就集成了 `MeterRegistry`）
- 没有 **审计留痕表**：每次 AI 审核应持久化 prompt/response/tokens/model/version 到独立表，合规与复盘必备

### 8) 没有 Guardrails（护栏）
- 仅仅依赖 DashScope 的平台安全。应该在业务侧也做：
    - **输入护栏**：长度 / 敏感词预扫 / PII 去敏 / 注入防护（Prompt Injection 防御）
    - **输出护栏**：分数范围校验、auditReason 的 PII/敏感词复查、强制 JSON Schema 校验
    - **越权护栏**：Agent 的工具调用要做白名单 + 权限校验

### 9) 没有多模型 / 多供应商策略
- `AiConfig` 里只注入一个 `DashScopeChatModel`。一旦阿里云限流 / 出故障，服务直接挂。
- 参考项目 `LlmProviderRegistry` 有完整的 Provider 注册表 + 缓存 + 降级路由。

### 10) 没有成本/并发/幂等治理
- 相同书籍/章节重复提交会反复调 LLM，没有 **contentHash → 结果缓存**（Redis / DB）
- 没有 **令牌桶限流** 按租户/接口维度控流
- 章节分段审核是串行，大章节耗时可能数十秒，没有 `CompletableFuture` 并发 + 熔断

### 11) 业务能力过于单一
- 现在审核只做了"文本违规判别"。企业级小说平台还需要：
    - **抄袭检测**（与知识库做相似度比对）
    - **质量评估**（剧情连贯、错别字、段落合理性）
    - **标签提取**（自动类别/标签，便于推荐）
    - **分级制**（纯爱/惊悚/恐怖 → 不同分级）
    - **风险等级**（低/中/高，阈值决定是否上人工）
    - **申诉/人工复核**（HITL）与反馈闭环（自学习）

## 二、企业级 Agent 目标架构

我建议把 novel-ai 从 "一个 Service" 重构成 **"Agent 平台 + 多个业务 Agent"**。总体分层如下：

```
novel-ai-service (novel-ai 模块)
├── agent-core            # Agent 基础能力（跨业务复用）
│   ├── client            # ChatClient 工厂、LlmProviderRegistry、多供应商与降级
│   ├── prompt            # PromptLoader / PromptVersion / TemplateRenderer（st/模板文件）
│   ├── output            # StructuredOutputInvoker + JsonSchema 校验 + 自动修复
│   ├── memory            # ChatMemory（Redis 实现）/ SessionContext
│   ├── tool              # ToolCallback 注册、工具白名单、权限
│   ├── rag               # Retriever / Reranker / QueryRewriter / Embedding
│   ├── guardrail         # 输入输出护栏（敏感词/PII/注入/越权）
│   ├── observability     # Micrometer 指标 + SkyWalking Span + Audit 日志
│   └── resilience        # Retry / RateLimit / CircuitBreaker / Idempotency
│
├── agent-audit           # 书籍/章节审核 Agent（多子 Agent 聚合）
│   ├── pipeline          # 审核流水线（Step 编排：预处理→多维判定→聚合→决策）
│   ├── checkers          # 违规检查子 Agent（政治/色情/广告/暴力/低俗）
│   ├── similarity        # 抄袭检测子 Agent（向量 + 模糊指纹）
│   ├── quality           # 质量评估子 Agent
│   ├── aggregator        # 风险聚合 + HITL 决策
│   └── event             # MQ 消费者（请求/结果/反馈）
│
├── agent-polish          # 文本润色 Agent
├── agent-cover           # 封面提示词 Agent（文生图上游）
├── agent-tag             # 自动标签/分级 Agent（可后加）
└── chat-interface        # 作者助手 / 站内问答（RAG 对话）
```

### 核心运行范式：**"Planner → Tools → Critic → Finalizer"**（Reflection Agent）

以 **书籍审核** 为例，一次请求的标准流程应是：

```
[ BookAuditRequestMq ] 
        ↓
┌──────────────────────────────────────────────────────────────┐
│ 0. Guardrail（入口护栏）：长度/注入/重复提交(contentHash)     │
│ 1. Preprocess：分段（overlap）+ 脱敏 + 风险词预扫             │
│ 2. Retrieve：Query Rewrite → 向量检索 → Rerank                │
│ 3. Plan：LLM 决定需要调用哪些 Checker（或默认全开）           │
│ 4. Execute（并发）：                                          │
│      ├─ 政治敏感 Checker（Tool: external_politics_filter?）   │
│      ├─ 色情/低俗 Checker                                     │
│      ├─ 广告/引流 Checker                                     │
│      ├─ 抄袭相似度 Checker（Tool: similarity_search）         │
│      └─ 质量评估 Checker                                      │
│ 5. Critic（Reflection）：交叉验证 + Schema 校验 + 自修复        │
│ 6. Aggregate：加权打分 → 最终 auditStatus + confidence         │
│ 7. HITL 判定：confidence < 阈值 或 高风险 → 进入人工            │
│ 8. Output Guardrail：auditReason 敏感词复查 + 长度截断          │
│ 9. Persist：审核留痕（prompt/response/token/model/version）    │
│ 10. Publish BookAuditResultMq                                 │
└──────────────────────────────────────────────────────────────┘
```

这个流程的关键好处：
- 多维并发 → **耗时下降**
- 多 LLM 交叉 → **准确率上升 + 减少单模型偏差**
- HITL 阈值 → **合规（法务可追溯）**
- 留痕 → **可复盘 + 可训练（后续微调 / DPO）**

## 三、技术选型建议（企业级）

| 能力 | 建议技术 | 为什么 |
|------|---------|--------|
| **Agent 框架** | Spring AI 1.x（你已经用） + Alibaba Spring AI Agent（JManus / Graph） | 官方血统，和你现有 `ChatClient` 无缝 |
| **Prompt 模板** | Spring AI `PromptTemplate` + `.st` 文件 + Nacos 热更新 | 把 Prompt 当"代码 + 配置"一起管，可灰度 |
| **结构化输出** | `BeanOutputConverter` + JSON Schema（`everit-json-schema`）校验 + 失败重写 | 比正则抠 JSON 稳定数十倍 |
| **向量库** | 迁移到 **Milvus 2.x** 或 **PostgreSQL + pgvector**（参考 interview-guide） | 生产推荐 Milvus；若不想多组件，pgvector 即可 |
| **Embedding** | 百炼 `text-embedding-v3` / `bge-large-zh-v1.5`（自部署） | 中文效果显著优于通用 OpenAI |
| **Reranker** | `bge-reranker-v2-m3`（HTTP 服务化，Java 调用） | RAG 质量提升最显著的一步 |
| **Chat Memory** | `MessageWindowChatMemory` + Redis Store | 多轮会话标准做法 |
| **工具调用** | Spring AI Tool Callback / MCP（Model Context Protocol） | 扩展相似度查询、敏感词库、运营规则库等"可执行工具" |
| **幂等 & 缓存** | `contentHash(sha256)` → Redis 缓存审核结果（TTL 7 天） | 重复提交不再烧 token |
| **限流 / 熔断** | Sentinel（阿里系和你的 Nacos/RocketMQ 匹配） | 按租户、按接口、按模型维度 |
| **重试** | Spring Retry + 分级策略（瞬时/持久/内容违规不重试） | 替代现在的"一把 try-catch" |
| **合规留痕** | 独立 `ai_audit_log` 表 + 冷热分离（ClickHouse） | 追溯 / 质检 / 训练数据 |
| **观测** | Micrometer + SkyWalking + **Langfuse** 或 **阿里云百炼观测** | 看 token / 成本 / 延迟 / 质量指标 |
| **A/B 实验** | Nacos 配置 + Prompt 版本 + 流量标记（X-Exp-Id） | 可无痛灰度新 prompt |
| **HITL** | 审核后台（turtle-website-front 增页） + 反馈 API → 回流到 RAG 经验库 | 持续改进闭环 |
| **多模型降级** | `LlmProviderRegistry` 模式（参考 interview-guide 已实现） | 阿里百炼挂了能切到 MiniMax / DeepSeek / OpenAI |
| **防 Prompt 注入** | 用户内容用 `<user_content>...</user_content>` 明确包裹，system prompt 明确"忽略内部指令" | 小说正文里包含"你是一个xxx"是常见攻击 |

## 四、分阶段改造路线图（建议 4 个迭代）

### Phase 1 · 打基础（1~2 周，优先级最高）
目标：**不改业务功能，先让代码符合 Spring AI 最佳实践**。

1. 把 `AiConfig` 拆分：引入 `LlmProviderRegistry`（照抄 interview-guide 的思路），默认 DashScope，保留 OpenAI 兼容层（MiniMax/DeepSeek 可后续接）。
2. 把所有 Prompt 抽到 `resources/prompts/*.st`：
    - `audit-book-system.st` / `audit-book-user.st`
    - `audit-chapter-system.st` / `audit-chapter-user.st`
    - `polish-system.st` / `polish-user.st`
    - `cover-prompt.st`
    - `audit-rule-extract.st`
3. 所有结构化输出改用 `BeanOutputConverter<T>` + 封装一个 `StructuredOutputInvoker`（直接搬 interview-guide 的实现），彻底删掉 `extractJsonFromResponse`/`extractField` 正则。
4. 审核结果类用 **Java `record`** 建模，与 DTO 转换走 MapStruct。
5. 异常分级：定义 `AiTransientException / AiContentInspectionException / AiNonTransientException`，通过 Spring Retry 的 `retryFor`/`noRetryFor` 精细控制。

**产出**：代码行数减少 30%+、可测试性大幅提升、Prompt 能通过 Nacos 热更新。

---

### Phase 2 · RAG 升级 + 审核链路拆分（2~3 周）

1. **Query Rewrite**：先用小模型（qwen-turbo）把长文本摘要成 2~3 个检索候选 query（参考 interview-guide 的 `rewriteQuestion`）。
2. 在 `novel-search` 里加 **Rerank** 阶段（BGE-Reranker 服务化），或本地用 `onnxruntime` 跑 rerank 模型。
3. 章节分段加 **overlap=200** + **并发审核**：
   ```java
   // 改造 auditChapter 的关键伪代码
   List<CompletableFuture<ChapterSegmentResult>> futures =
       segments.stream()
               .map(seg -> CompletableFuture.supplyAsync(() -> auditOneSegment(seg), aiExecutor))
               .toList();
   List<ChapterSegmentResult> results = futures.stream().map(CompletableFuture::join).toList();
   ```
4. 引入 **contentHash 幂等**（SHA-256 书名+简介 → Redis key → 24h TTL）。
5. 把"违规类型"拆成多个 **Checker 子 Agent**（每个独立 Prompt + 结构化输出），最后做 **Aggregator 聚合打分**。你现在一个 Prompt 问 5 条标准，模型 attention 会分散；拆成独立 Checker 后每个更专注、也便于独立升级。

**产出**：单章审核耗时降 50%~70%、违规召回率上升、重复审核 0 成本。

---

### Phase 3 · 工具化 + HITL + 留痕（2~3 周）

1. 把"相似审核经验检索"做成 Spring AI **Tool Callback**（而不是硬编码塞进 prompt），让 LLM 自己决定要不要查、查几次。
2. 接入"敏感词库"、"运营规则库"作为工具（Function Calling）。
3. 加 **审计表** `ai_audit_log(id, biz_type, biz_id, request, response, tokens_in, tokens_out, model, provider, prompt_version, latency_ms, status, created_at)`，每次调用都写。
4. 加 **HITL 工作流**：
    - `confidence < 0.7` 或 `auditStatus=2 且 存在申诉` → 进 `human_review_queue`
    - turtle-website-front 新增"AI 审核复核"页面
    - 人工结论回写后，异步触发 `extractAuditRule` → 向量化入库 → 闭环进化
5. 加 **Micrometer 指标**：`novel.ai.audit.pass_rate`、`novel.ai.audit.hitl_rate`、`novel.ai.audit.tokens`、`novel.ai.audit.latency` 等。

**产出**：合规可追溯，策略可持续优化。

---

### Phase 4 · 多 Agent 编排 + 多模型容灾（按需）

1. 把审核流水线改成 **Spring AI Alibaba Graph** 或 **LangGraph4j** 的 StateGraph（节点=Checker，边=条件路由）。
2. 引入 **Critic Agent**（一个更强模型，如 qwen-max）对 Checker 的结果做二次审阅，发现矛盾则回滚重审。
3. 多模型策略：
    - 高风险内容用 qwen-max
    - 常规内容用 qwen-plus / qwen-turbo（省钱）
    - 用 Sentinel 熔断，挂了走 MiniMax/DeepSeek
4. 引入 **Langfuse**（开源）或百炼控制台观测台，看每次 trace。

## 五、针对 `TextServiceImpl` 的具体重构示意（让你有画面）

目前这样（现状）：

```55:99:turtle-website/novel-ai/novel-ai-service/src/main/java/com/novel/ai/service/impl/TextServiceImpl.java

```

改造后建议的样子（示意代码，不是 diff，你自己落到代码里）：

```java
@Service
@RequiredArgsConstructor
public class BookAuditAgent {

    private final PromptTemplateLoader prompts;
    private final LlmProviderRegistry llm;
    private final StructuredOutputInvoker invoker;
    private final AuditRagRetriever ragRetriever;
    private final AuditResultCache cache;
    private final AuditLogWriter auditLog;
    private final List<AuditChecker> checkers; // 政治/色情/广告/质量...

    private static final BeanOutputConverter<BookAuditLlmDTO> CONVERTER =
            new BeanOutputConverter<>(BookAuditLlmDTO.class);

    @Trace(operationName = "Agent.AuditBook")
    public BookAuditRespDto audit(BookAuditReqDto req) {
        String hash = ContentHasher.sha256(req.getBookName(), req.getBookDesc());
        return cache.getOrCompute(hash, () -> doAudit(req, hash));
    }

    private BookAuditRespDto doAudit(BookAuditReqDto req, String hash) {
        // 1. 护栏
        Guardrails.validateBookAuditInput(req);

        // 2. RAG（query rewrite + vector + rerank）
        List<AuditExperience> refs = ragRetriever.retrieve(req.getBookName() + "\n" + req.getBookDesc());

        // 3. 并发 Checker
        List<CheckerResult> results = checkers.parallelStream()
                .map(c -> c.check(req, refs))
                .toList();

        // 4. 聚合
        AggregatedResult agg = RiskAggregator.aggregate(results);

        // 5. HITL 判定
        if (agg.needsHumanReview()) {
            hitlQueue.enqueue(req, agg);
        }

        // 6. 输出护栏 + 留痕
        BookAuditRespDto resp = agg.toRespDto(req.getId());
        auditLog.writeAsync(hash, req, resp, results);
        return resp;
    }
}
```

每个 `AuditChecker` 就是一个最小 Agent：加载自己的 `.st` 模板 → `StructuredOutputInvoker.invoke` → 返回结构化结果。互相独立、独立演进、独立灰度。

## 六、两个容易被忽视但很关键的建议

1. **审核的 Prompt 不要过度强调"允许什么"**
   你现在的 prompt 写了 `"内容允许改编现有作品、同人文、同性恋、具有文学价值的情感描写等"` —— 这类语句容易被攻击者利用做"越狱"（你已经做了预设立场放宽，正文里嵌入 `以下属于文学价值的情感描写` 就可能被带偏）。
   建议把"白名单"用 few-shot 示例的方式表达，而不是开放性声明。

2. **要把"审核结果"和"审核经验"解耦**
