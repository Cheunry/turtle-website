# 架构设计说明（初稿）

本文档用于说明项目整体设计，重点回答 4 个问题：
1. 为什么这么拆服务？
2. 为什么选择这些技术？
3. 出问题时怎么排查？
4. 做过哪些优化，效果如何？

## 1. 系统目标与边界

- **目标**：构建一个可扩展、可观测、支持高并发读写与搜索的小说平台后端。
- **核心业务**：用户与作者体系、书籍与章节管理、内容审核、搜索检索、消息通知等。
- **系统边界**：本仓库聚焦后端微服务；前端工程独立维护。

## 2. 微服务拆分与职责

### 2.1 网关服务 `novel-gateway`

- 对外统一入口，负责路由转发、统一日志、跨服务入口治理。
- 统一处理鉴权相关上下文（依赖后端服务完成用户信息校验）。
- 基于 Spring Cloud Gateway + WebFlux，支撑高并发和长耗时请求场景。

### 2.2 用户服务 `novel-user`

- 用户登录注册、用户资料、作者身份相关能力。
- 承担部分创作链路入口（接收请求后通过 MQ 异步投递给下游处理）。

### 2.3 书籍服务 `novel-book`

- 书籍信息、章节信息、审核状态、核心业务规则。
- 消费 MQ 消息并落库，触发后续搜索同步与AI审核链路。

### 2.4 搜索服务 `novel-search`

- 承担检索能力：关键词检索 + 向量语义检索（混合搜索）。
- 与 Elasticsearch 交互，消费书籍变更事件以保持索引实时更新。

### 2.5 AI 服务 `novel-ai`

- AI 相关能力封装（例如内容审核与模型调用链路）。
- 通过服务调用或消息驱动与业务服务协同。

### 2.6 公共模块 `novel-core`

- 提供公共常量、通用工具、跨服务复用能力。
- 减少重复代码，统一跨模块约束。

## 3. 技术选型与理由

## 3.1 微服务框架：Spring Boot + Spring Cloud + Alibaba

- 生态成熟，和 Nacos、Gateway、OpenFeign、RocketMQ 配套完整。
- 适合业务快速拆分、独立部署、按服务演进。

## 3.2 配置与注册中心：Nacos

- 同时支持配置中心 + 服务发现，减少中间件种类。
- 动态配置能力强，适合多环境与线上快速调整。

## 3.3 异步解耦：RocketMQ

- 把高耗时链路从同步请求中剥离，降低接口 RT。
- 用于“提交请求后异步处理”的核心业务场景（如书籍/章节处理）。

## 3.4 搜索：Elasticsearch + 向量检索

- BM25 解决关键词精准匹配；向量检索提升语义召回。
- 混合搜索兼顾准确率与覆盖率，适合小说场景长尾查询。

## 3.5 可观测性：SkyWalking

- 支持微服务链路追踪，便于定位跨服务性能瓶颈。
- 与容器部署结合，适合线上快速排障。

## 4. 核心业务链路（面试可直接讲）

### 4.1 创作链路（作者发布/修改内容）

1. 请求从网关进入用户服务。  
2. 用户服务完成轻量校验后发送 MQ 消息并快速返回。  
3. 书籍服务异步消费消息，执行落库、业务规则校验与状态流转。  
4. 后续触发搜索同步与审核相关处理。  

**收益**：入口 RT 降低，跨服务同步依赖减少，系统并发能力更稳定。

### 4.2 搜索链路（读者检索）

1. 用户请求进入搜索服务。  
2. 搜索服务并行执行 BM25 检索与向量检索。  
3. 对两路结果进行融合排序后返回。  

**收益**：既支持关键词精确命中，也支持语义相关召回。

## 5. 出问题怎么排查（总流程）

建议按“入口 -> 服务 -> 中间件 -> 数据层”逐层定位：

1. **看入口**：网关日志与状态码，确认是否已进入系统。  
2. **看链路**：SkyWalking 追踪调用耗时与失败节点。  
3. **看消息**：RocketMQ 是否堆积、消费者是否异常。  
4. **看配置**：Nacos 配置是否生效（环境/命名空间/分组）。  
5. **看数据**：MySQL/ES/Redis 是否数据不一致或延迟。  

详细故障案例可参考：`docs/faq.md` 与 `doc/note/NACOS_TROUBLESHOOTING.md`。

## 6. 已做优化（阶段性总结）

当前已沉淀的优化方向：

- **同步转异步**：部分作者侧操作改为 MQ 驱动，减少同步 Feign 链路。
- **缓存优化**：引入 Redis 缓存用户/作者信息，降低重复查询。
- **搜索增强**：构建 ES + 向量混合搜索，提升检索体验。
- **链路可观测**：引入 SkyWalking，支持性能问题快速定位。

效果与数据详情见：`doc/note/性能优化总结文档.md`。

## 7. 当前风险与后续完善点

- 配置中心依赖高：Nacos 不可用时影响服务启动与发现。
- 文档与代码同步成本高：需要把关键链路变更纳入版本记录。
- 建议逐步补齐：
  - 各模块时序图
  - 关键表结构说明
  - 统一错误码与重试策略文档

## 8. 网关模块详解（面试可直接讲）

### 8.1 为什么网关单独拆服务

- **统一入口**：前端与外部调用只访问网关，避免直接暴露内部服务。
- **统一治理**：路由、跨域、日志、鉴权等能力集中处理，减少重复实现。
- **隔离变化**：业务服务内部重构时，只要路由契约不变，外部调用方无感知。

### 8.2 选型与取舍

- **选型**：Spring Cloud Gateway（未使用 Zuul）。
- **理由**：
  - 基于 WebFlux/Netty 的非阻塞模型，更适合高并发与长耗时请求。
  - 与 Spring Cloud 2024.x、Nacos、LoadBalancer 生态集成更自然。
  - 过滤器链机制清晰，便于统一扩展日志、鉴权、限流等能力。

### 8.3 当前实现（基于现有代码）

- **启动入口**：`NovelGatewayApplication`。
- **核心依赖**：
  - `spring-cloud-starter-gateway`
  - `spring-cloud-starter-alibaba-nacos-discovery`
  - `spring-cloud-starter-alibaba-nacos-config`
  - `spring-cloud-starter-loadbalancer`
- **全局日志过滤器**：`LoggingFilter`
  - 前置记录 `Path/Method/Headers/QueryParams`
  - 后置记录响应状态码
  - 优先级为 `HIGHEST_PRECEDENCE`，用于覆盖完整请求生命周期
- **跨域配置**：`NovelCorsConfig`
  - 统一放行请求头/方法
  - 允许携带凭证（Cookie）
  - 当前使用 `allowedOriginPattern("*")`，开发联调方便，但生产建议收敛域名白名单
- **容器端口映射**：生产编排中 `8080 -> 8888`（外部访问 8080）

### 8.4 请求流转

1. 客户端请求进入网关（`8080`）。  
2. `LoggingFilter` 记录请求基础信息。  
3. 路由断言命中目标服务（路由规则通常托管于 Nacos 配置）。  
4. 通过 `lb://` 协议做服务发现与负载均衡。  
5. 下游服务返回后，网关记录响应状态并回传客户端。  

### 8.5 网关维度排障 SOP

1. **先看入口是否通**：端口、容器、健康状态。  
2. **再看路由是否命中**：网关日志里是否出现目标路径。  
3. **再看服务发现**：Nacos 是否有目标实例、实例是否健康。  
4. **再看配置是否生效**：环境、分组、命名空间是否一致。  
5. **最后看下游错误**：确认 5xx 是网关层问题还是业务服务透传。  

### 8.6 常见问题

- **Nacos 连接异常**：会导致网关无法发现下游服务，表现为转发失败/5xx。
- **跨域问题**：浏览器报 CORS 错误通常优先检查网关 CORS 配置。
- **下游超时**：长耗时请求若超时参数不匹配，可能出现 504。
- **日志噪声过大**：全量打印 Header/Query 在高峰期可能增加 IO 压力。

### 8.7 后续优化计划（建议）

1. **鉴权前置增强**：补全统一 JWT 解析与细粒度白名单策略。  
2. **限流与熔断**：在网关层增加 IP/用户维度限流，保护下游服务。  
3. **日志脱敏与采样**：避免敏感头信息泄漏，降低高并发下日志成本。  
4. **路由变更规范化**：固定 Nacos 路由配置模板，减少人为误配。  
5. **可观测性增强**：打通网关指标（QPS、错误率、P95/P99）到统一监控面板。
