# API 文档说明（初稿）

当前项目是微服务架构，接口文档建议按“网关统一入口 + 服务内分组”维护。

## 1. 对外入口规则

- 对外统一从 `novel-gateway` 进入。
- 前端与第三方调用不直接访问内部服务端口。
- 鉴权采用 JWT（统一鉴权策略）。

## 2. 文档组织建议

建议把接口分为四组：

1. **用户与认证接口**（用户服务）  
2. **作者与创作接口**（用户服务 + 书籍服务链路）  
3. **书籍与章节接口**（书籍服务）  
4. **搜索接口**（搜索服务）  

## 3. 接口说明模板（建议后续统一）

每个接口至少包含：

- 接口名
- 请求路径
- 请求方法
- 是否鉴权
- 请求参数
- 返回结构
- 业务约束（例如状态机、权限、幂等等）
- 常见错误码

示例模板：

```md
### 获取书籍详情
- Path: `/api/front/book/{id}`
- Method: `GET`
- Auth: 否
- 请求参数：
  - `id` (Long) 书籍ID
- 返回：
  - `code`
  - `message`
  - `data`（书籍基础信息）
- 备注：
  - 仅返回审核通过状态的数据（示例，按真实逻辑调整）
```

## 4. 当前阶段建议

- 第一阶段：先梳理“高频 + 面试高价值”接口（登录、发布书籍、发布章节、搜索）。
- 第二阶段：补齐请求/响应示例与错误码。
- 第三阶段：补齐时序图（接口触发 MQ、异步处理节点）。
